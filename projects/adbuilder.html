Projet lab Active Directory

ADBuilder

Moteur PowerShell déclaratif pour construire un lab Active Directory depuis un JSON : promotion AD DS, reprise après reboot, providers Directory, DNS, GPO, KDS, dMSA, AD CS, assertions finales et compilation de scénarios vulnérables reproductibles pour l'entraînement offensif.

Note publique : ADBuilder est prévu pour des VM jetables. Il configure Windows Server depuis l'intérieur de la machine, promeut le serveur en contrôleur de domaine, applique les providers activés, puis écrit des rapports exploitables pour vérifier ce qui a réellement été construit.

1. Objectif

ADBuilder remplace la construction manuelle de labs Active Directory par une configuration déclarative. Le JSON décrit le mode du domaine, la stratégie de réconciliation, les providers à activer, les objets attendus et les assertions finales. Le moteur lit ce contrat, le valide, calcule l'ordre d'exécution puis applique les changements de façon reprenable.

L'outil sait créer un nouveau forest, ajouter un contrôleur de domaine à un domaine existant ou préparer un child domain. La cible principale reste le lab isolé sur Windows Server Core 2019, 2022 ou 2025. Les fonctionnalités dMSA demandent Server 2025.

La partie intéressante n'est pas seulement la création d'objets AD. ADBuilder construit aussi des scénarios offensifs contrôlés : ACL dangereuses, délégations, Kerberoast, AS-REP roast, RBCD, DCSync, AD CS ESC, dMSA et GPO faibles. Chaque scénario est décrit haut niveau puis compilé en objets, ACLs, templates ou migrations concrètes.

2. Surface JSON

Le fichier racine contient metadata, mode, execution et providers. Les modes supportés sont newForest, additionalDC et childDomain. Les providers actuels sont directory, dns, gpo, kds, dmsa, adcs et assertions.

La configuration Directory couvre les sites, subnets, site links, OUs, groupes, utilisateurs, ordinateurs, memberships, FGPP, délégations et ACL edges. Les autres providers ajoutent les enregistrements DNS, les GPO avec registry/security filtering/permission edges, la racine KDS, les comptes dMSA, les migrations dMSA, l'installation d'une Enterprise CA, les templates AD CS et les assertions finales.

3. Workflow

Le point d'entrée normal est RUN-ME.ps1. Il prépare la VM si nécessaire, lance les validations, exécute un dry-run, démarre la Stage A, puis reprend la Stage B après reboot. On relance la même commande après chaque redémarrage jusqu'au message final Done. 

# Configuration sûre de référence
.\RUN-ME.ps1 -StaticIP 172.20.10.50 -Gateway 172.20.10.1 -ComputerName DC01 -ConfigPath .\examples\reference.json

# Configuration volontairement vulnérable
.\RUN-ME.ps1 -StaticIP 172.20.10.50 -Gateway 172.20.10.1 -ComputerName DC01 -ConfigPath .\examples\adcs-esc1-lab.json -LabUnsafe
VM Windows Server Core jetable
    -> préparation réseau, DNS, hostname et secrets
    -> test négatif des modes dangereux
    -> validation JSON canonique avec NJsonSchema
    -> validation sémantique ADBuilder
    -> dry-run des providers
    -> Stage A : installation AD DS + promotion
    -> reboot
    -> Stage B : attente ADWS / SYSVOL / NETLOGON / DNS
    -> providers exécutés dans l'ordre du DAG
    -> assertions finales
    -> ADBuilder-Report.json + contrat dMSA

Le DAG évite l'ordre d'exécution fragile codé à la main. Chaque provider déclare ce dont il dépend et ce qu'il produit. Le state file garde les phases déjà complétées, le hash de config et le hash de config compilée, ce qui permet de reprendre proprement ou de bloquer un drift non assumé. 

domain.ready
    -> directory
        -> directory.ousReady
        -> directory.principalsReady
        -> directory.aclsReady
    -> dns
    -> gpo
    -> kds
        -> kds.rootKeyReady
    -> dmsa
        -> dmsa.accountsReady
    -> adcs
        -> adcs.configured
    -> assertions

4. Scénarios vulnérables

La section labVulnerabilities sert de compilateur de scénarios. Au lieu d'écrire toutes les ACLs et tous les objets à la main, on décrit l'abus voulu, puis ADBuilder injecte les objets Directory, ACL edges, templates AD CS, permissions GPO ou migrations dMSA nécessaires. 

{
  "labVulnerabilities": {
    "enabled": true,
    "items": [
      {
        "id": "ACL_FORCE_CHANGE_PASSWORD",
        "labUnsafe": true,
        "trustee": "Helpdesk",
        "target": "victim.user"
      },
      {
        "id": "ADCS_ESC1",
        "labUnsafe": true,
        "enrollers": ["ESC1-Enrollers"]
      }
    ]
  }
}

Les scénarios supportés couvrent notamment GenericAll, WriteDacl, WriteOwner, AddMember, ForceChangePassword, Shadow Credentials, RBCD, DCSync, Kerberoast, AS-REP roast, délégation contrainte et non contrainte, AD CS ESC1/ESC2/ESC3/ESC4/ESC7, GPO weak ACL et plusieurs variantes dMSA post-patch ou legacy.

5. Garde-fous

  • Lab isolé : ADBuilder est pensé pour des VM dédiées, snapshots et domaines jetables.
  • -LabUnsafe explicite : KDS root key, dMSA, Enterprise CA, templates vulnérables, Deny ACEs, ACL dangereuses et labVulnerabilities sont bloqués sans opt-in.
  • Validation en deux couches : schéma JSON canonique, puis validation sémantique propre au domaine ADBuilder.
  • Secrets hors JSON : mots de passe via environnement, prompt ou SecretManagement.
  • Réconciliation contrôlée : modes createOnly, additive, update et exact, avec suppressions, déplacements et resets de mot de passe soumis à des flags explicites.
  • Drift détecté : la reprise après reboot compare les hashes et demande -IgnoreDrift si la config a changé.
  • Rapports de fin : ADBuilder écrit un rapport JSON et un contrat dMSA listant les objets et ACLs configurés.

6. Limites

  • L'outil ne provisionne pas l'hyperviseur : il agit depuis l'intérieur d'une VM Windows Server déjà créée.
  • Les fonctionnalités dMSA visent Windows Server 2025. Sur 2019/2022, la surface Directory, DNS, GPO, assertions et une partie AD CS restent les usages principaux.
  • Les features AD CS et KDS sont forest-wide. Elles doivent rester dans des environnements de lab.
  • Le cleanup exige un runId confirmé et -LabUnsafe. Ce n'est pas un mécanisme de rollback de production.
  • La qualité du lab dépend du JSON : le fichier reste le contrat et la source de vérité.

7. Ce que cela démontre

ADBuilder montre une compétence qui complète directement le pentest Active Directory : savoir construire des environnements réalistes, répétables et vérifiables, pas seulement les attaquer. Le projet force à comprendre AD DS, Server Core, les objets Directory, les ACLs, les GPO, AD CS, dMSA, KDS, DNS et les chemins d'exploitation qu'on veut reproduire.

Côté engineering, le projet combine module PowerShell, JSON Schema, validation sémantique, orchestration par DAG, state management, reprise après reboot, dry-run, reporting et tests. Le résultat est un générateur de labs versionnables : on peut reconstruire un scénario depuis une VM fraîche, le partager, le casser, puis le recréer proprement.
