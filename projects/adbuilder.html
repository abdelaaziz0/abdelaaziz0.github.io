Projet lab Active Directory

ADBuilder

Moteur PowerShell complet pour construire automatiquement un environnement Active Directory de lab depuis un fichier JSON : promotion du contrôleur de domaine, création des objets AD, configuration des groupes, utilisateurs, ordinateurs, politiques de mots de passe, GPO, Kerberos, DNS, AD CS et vérifications finales.

Note publique : ce projet est conçu pour créer des environnements Active Directory de lab reproductibles. Il automatise la construction du domaine et des composants associés dans une VM isolée, avec une logique de validation, de reprise après reboot et de vérification finale.

1. Objectif

ADBuilder génère un lab Active Directory complet à partir d'un fichier JSON. Le but est simple : remplacer les manipulations manuelles longues et fragiles par une configuration déclarative, claire et rejouable. Le fichier JSON devient la source de vérité du lab : il décrit le domaine, les unités d'organisation, les groupes, les utilisateurs, les ordinateurs, les memberships, les politiques de mots de passe fines, les GPO, les paramètres Kerberos, les entrées DNS, les éléments AD CS et les assertions attendues.

Le projet fonctionne sur Windows Server Core avec PowerShell 5.1. Il prend en charge la promotion du serveur en contrôleur de domaine, le reboot obligatoire après promotion, puis la reprise automatique pour appliquer le reste de la configuration. Cette séparation en deux phases donne un déploiement propre, contrôlé et adapté au fonctionnement réel d'Active Directory.

Le résultat est un environnement AD prêt à l'emploi pour des scénarios de formation, de démonstration, de CTF ou de recherche interne. Le lab peut être reconstruit depuis une VM fraîche, partagé avec un autre opérateur et vérifié automatiquement à la fin du déploiement.

2. Schéma JSON

Le JSON décrit l'état attendu du lab. Il contient les paramètres du domaine, les providers à activer et les objets à créer. Le moteur lit cette configuration, la valide, résout les dépendances entre composants, puis applique les changements dans le bon ordre.

3. Workflow

Le workflow suit le comportement réel d'un contrôleur de domaine Windows. Le déploiement est séparé en deux phases : une première phase prépare la machine et lance la promotion AD DS, puis une seconde phase reprend après reboot pour appliquer la configuration complète du lab. 

VM Windows Server Core fraîche
    -> préparation réseau et hostname
    -> génération ou chargement des secrets de lab
    -> validation du JSON
    -> dry-run
    -> Stage A : installation AD DS + promotion du domaine
    -> reboot
    -> Stage B : attente ADWS / SYSVOL / NETLOGON / DNS
    -> exécution des providers dans l'ordre du DAG
    -> assertions finales
    -> résumé du déploiement

Le script principal peut être lancé depuis une VM fraîche. Il configure la machine, applique les paramètres réseau, prépare les secrets, lance la promotion du domaine, puis reprend automatiquement après reboot. 

.\RUN-ME.ps1 -StaticIP 6.150.5.50 -Gateway 6.150.5.1 -ComputerName DC01

L'ordonnancement des composants est piloté par un DAG. Chaque provider déclare les capacités dont il dépend et celles qu'il produit. Cette approche évite un ordre d'exécution fragile et construit le lab dans une séquence cohérente. 

domain.ready
    -> directory
        -> directory.ousReady
        -> directory.principalsReady
    -> dns
    -> kerberos
    -> gpo
    -> adcs
    -> assertions

À la fin du run, ADBuilder vérifie l'état du lab : existence des utilisateurs, groupes, ordinateurs, memberships, politiques FGPP, objets AD CS et relations attendues. Le lab est créé, puis validé.

4. Garde-fous

  • Lab isolé : l'outil est prévu pour des VM dédiées, snapshots et environnements contrôlés.
  • Secrets séparés du JSON : les mots de passe sont fournis par variables d'environnement ou générés par le script de préparation.
  • Validation avant exécution : le fichier JSON est vérifié avant toute modification du domaine.
  • Dry-run : le plan peut être simulé avant application réelle.
  • État de run : un fichier d'état reprend proprement après reboot.
  • Assertions finales : les objets et relations attendus sont contrôlés automatiquement.
  • Reconciliation additive : le comportement par défaut crée et complète le lab sans suppression destructive non demandée.

5. Limites

  • L'outil est destiné aux environnements de lab et ne doit pas être lancé sur un domaine de production.
  • Il ne crée pas les machines virtuelles ni l'infrastructure hyperviseur : il configure Windows Server et le domaine depuis l'intérieur de la VM.
  • Les opérations destructives sont volontairement limitées pour éviter de supprimer des objets AD par erreur.
  • Les secrets générés doivent être protégés puis supprimés après utilisation du lab.
  • La qualité du lab dépend de la précision du JSON : le fichier de configuration reste la source de vérité.

6. Ce que cela démontre

Ce projet démontre une compétence complémentaire au pentest Active Directory : la capacité à construire un environnement maîtrisé, reproductible et vérifiable. ADBuilder ne se limite pas à créer quelques objets AD ; il automatise un cycle complet de lab, depuis la préparation de Windows Server Core jusqu'aux assertions finales.

Techniquement, le projet couvre PowerShell avancé, Active Directory Domain Services, promotion de domaine, gestion d'état après reboot, Server Core, validation JSON, ordonnancement par DAG, gestion des secrets, politiques de mot de passe fines, memberships d'ordinateurs, GPO, Kerberos, DNS et AD CS.

L'intérêt principal est la reproductibilité : un même scénario peut être reconstruit depuis une VM fraîche, partagé, versionné et amélioré. C'est une base solide pour créer des labs de formation AD, des scénarios de démonstration ou des environnements CTF centrés sur Active Directory.
