Projet Active Directory

ntds2bh

Outil d'analyse offline qui transforme un fichier NTDS.dit autorisé en archive ZIP compatible BloodHound CE. Il reconstruit les objets, relations, ACL, chemins de délégation, éléments AD CS et trusts afin d'étudier la structure réelle d'un domaine Active Directory sous forme de graphe.

1. Objectif

Un dump NTDS.dit ne contient pas seulement des secrets ou des anciens hash. Il contient aussi la structure profonde du domaine : groupes privilégiés, memberships, ACL, objets GPO, trusts, chemins de délégation, certificats, templates AD CS et relations entre objets. Même lorsqu'un dump est ancien, cette structure reste souvent utile car l'architecture Active Directory évolue généralement moins vite que les mots de passe.

ntds2bh exploite cette valeur structurelle. L'outil prend un NTDS.dit offline, le parse, reconstruit les objets et relations utiles, puis génère une archive bloodhound.zip directement importable dans BloodHound CE. L'objectif est de passer d'une base AD brute à un graphe exploitable, sans dépendre d'une collecte LDAP live.

Le projet ne cherche pas à remplacer l'analyse humaine. Il automatise la reconstruction du graphe pour que l'auditeur lise plus vite les chemins de privilèges, les relations dangereuses, les ACL sensibles, les délégations Kerberos, les trusts et les éléments AD CS exposés.

2. Entrées et sorties

Élément Description
Entrée principale Fichier NTDS.dit obtenu dans un cadre autorisé, par exemple depuis un lab ou une copie offline issue d'un audit interne.

3. Architecture

NTDS.dit est une base ESE/JET Blue. Elle n'est pas structurée comme un export LDAP simple. Les objets, attributs, relations et security descriptors sont répartis dans plusieurs tables internes. Les attributs utilisent des colonnes ATT*, les noms distingués doivent être reconstruits à partir de DNT_col, PDNT_col, RDN et RDNtyp_col, et les ACL sont stockées sous forme de security descriptors binaires.

L'outil reconstruit progressivement une vue exploitable du domaine. Les tables les plus importantes sont :

  • datatable : objets AD et majorité des attributs.
  • link_table : relations liées, dont memberships et backlinks.
  • sd_table : security descriptors binaires référencés par les objets.
NTDS.dit
    -> ouverture ESE
    -> lecture datatable / link_table / sd_table
    -> reconstruction du schéma et des DNs
    -> normalisation des objets
    -> résolution des SIDs, ACL, memberships, trusts et délégations
    -> mapping vers les modèles BloodHound CE
    -> génération de bloodhound.zip

Les deux premières étapes reposent sur une base de parsing héritée de ntdissector. La couche propre à ntds2bh intervient ensuite pour transformer les données NTDS normalisées en objets et relations BloodHound CE.

Voir le schéma Mermaid du pipeline global 
flowchart LR
    ntds[("NTDS.dit")]:::input
    parse["ESE parse<br/>schema + DN rebuild<br/>record normalization"]:::process
    cache["caches<br/>SIDs, DNs, links<br/>security descriptors"]:::cache
    route{"route by class<br/>or sAMAccountType"}:::decision
    map["map to BloodHound<br/>users, computers, groups<br/>OUs, GPOs, ADCS, trusts<br/>ACLs -> edges"]:::process
    zip[["bloodhound.zip"]]:::output

    ntds --> parse
    parse --> cache
    cache --> route
    route --> map
    map --> zip

4. Problèmes techniques

Security descriptors et ACL

Les objets AD ne contiennent pas directement une ACL prête à consommer. Ils portent une référence vers un security descriptor stocké dans sd_table. Pour produire des relations BloodHound, l'outil doit récupérer le blob binaire, parser l'en-tête, extraire le owner SID, le group SID, la DACL, les ACEs, puis traduire les droits Windows en relations compréhensibles par BloodHound.

Cette étape est centrale : BloodHound n'a pas besoin d'une ACL brute, mais d'arêtes comme Owns, GenericAll, GenericWrite, WriteDacl, WriteOwner, AddMember ou des droits liés à DCSync et AD CS. ntds2bh effectue cette conversion en tenant compte des masques d'accès, GUIDs d'objet, ACEs héritées, ACEs refusées, SID spéciaux et types d'objets.

Voir le pipeline de résolution des security descriptors 
record.nTSecurityDescriptor
    -> sd_id reference
    -> sd_table.sd_value
    -> parse_ntSecurityDescriptor()
    -> owner SID / group SID / DACL / SACL
    -> parse_aces()
    -> resolve_aces()
    -> BloodHound Aces[]
       - PrincipalSID
       - PrincipalType
       - RightName
       - IsInherited

Relations AD non triviales

Les relations Active Directory ne sont pas toutes stockées comme des attributs directs. Les memberships de groupe résident dans link_table. Le primaryGroupID d'un utilisateur ou d'un ordinateur doit être résolu séparément puis réinjecté dans la liste des memberships. Les délégations Kerberos passent par des SPNs qu'il faut faire correspondre à des comptes réels. Les well-known SIDs doivent être normalisés pour être acceptés par BloodHound. Les trusts sont stockés dans des objets trustedDomain et doivent être rattachés au bon domaine.

Cette reconstruction est ce qui donne sa valeur au projet : sans ces étapes, l'import BloodHound produirait des nœuds isolés ou des relations incomplètes. Avec cette logique, le dump offline redevient un graphe analysable.

5. Validation

Les sorties sont validées sur des environnements de lab contrôlés avant import dans BloodHound CE. Les contrôles portent sur la présence des objets attendus, la cohérence des SIDs, la résolution des distinguished names, les memberships, les trusts, les relations de délégation, les ACL dérivées et les objets AD CS.

L'objectif de validation est simple : un objet présent dans NTDS.dit doit être routé vers la bonne collection BloodHound, et une relation présente dans la base doit produire l'arête correspondante dans le graphe. Les doublons, relations orphelines, SIDs non résolus et mappings ambigus sont traités pendant la normalisation.

6. Limites

  • L'outil travaille à partir d'un snapshot offline. Il reconstruit l'état contenu dans le fichier fourni, pas l'état live actuel du domaine.
  • La qualité du graphe dépend de la qualité et de l'exhaustivité du NTDS.dit fourni.
  • Un graphe BloodHound accélère l'analyse, mais les chemins critiques doivent toujours être validés par l'auditeur.
  • La page publique décrit l'architecture et le fonctionnement général sans exposer de dump réel, donnée client ou secret.

7. Ce que cela démontre

Ce projet démontre une compétence avancée en analyse Active Directory : lire une base NTDS offline, reconstruire les objets du domaine, convertir des security descriptors binaires en relations exploitables et produire un artefact directement utilisable dans BloodHound CE.

Il montre aussi une compréhension concrète des structures internes AD : tables ESE, schéma, DNT, distinguished names, SIDs, group memberships, primary groups, ACL, ACEs, trusts, délégation Kerberos et objets AD CS. L'intérêt n'est pas seulement de parser des données, mais de reconstruire les relations qui rendent l'analyse du domaine lisible.

Pour un audit ou un lab, ntds2bh transforme un dump difficile à exploiter manuellement en graphe lisible, navigable et partageable. C'est un pont entre l'analyse bas niveau d'un NTDS.dit et la visualisation orientée chemins de privilèges de BloodHound.

8. Attribution technique

Le parsing NTDS bas niveau reprend des composants hérités de ntdissector, dont l'ouverture de la base ESE, la lecture des tables internes, la résolution du schéma et la normalisation des attributs. ntds2bh ajoute la couche orientée BloodHound : typage des SIDs, résolution des ACL, mapping des ACEs vers des droits BloodHound, reconstruction des relations, gestion des objets AD CS et écriture de l'archive finale.

Voir la répartition détaillée du code 
Hérité de ntdissector :
- ouverture de la base ESE via dissect.database.ese
- lecture de datatable, link_table et sd_table
- résolution des colonnes ATT* vers les noms LDAP
- résolution des objectClass
- reconstruction des distinguished names
- sérialisation et formatage des records NTDS

Ajouté par ntds2bh :
- get_parsed_sd() pour retrouver les security descriptors depuis sd_table
- build_sid_to_type() pour typer les SIDs et préparer les caches
- utils/ace_resolver.py pour mapper les ACEs vers les droits BloodHound
- utils/sid_resolver.py pour gérer well-known SIDs et types d'objets
- utils/adcs.py pour normaliser les objets AD CS
- pipeline/orchestrator.py pour router les objets et reconstruire les relations
- models/*.py pour produire les shapes BloodHound CE
- pipeline/writer.py pour écrire bloodhound.zip
