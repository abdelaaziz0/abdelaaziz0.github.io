04 - Expired Passwords

Expired Passwords & Force-Change

What pointed us here The incident.txt file on the share said that accounts with weak passwords had been expired. An empty-password spray confirmed two accounts respond with STATUS_PASSWORD_MUST_CHANGE instead of STATUS_LOGON_FAILURE. 
nxc smb dc.sendai.vl -u users.txt -p '' --continue-on-success

[-] sendai.vl\elliot.yates:  STATUS_PASSWORD_MUST_CHANGE
[-] sendai.vl\thomas.powell: STATUS_PASSWORD_MUST_CHANGE
STATUS_PASSWORD_MUST_CHANGE - forcer le changement sans l'ancien mot de passe

Quand un compte AD a l'attribut pwdLastSet = 0 (password expired/must change), Windows autorise une authentification partielle : l'utilisateur peut se connecter aux protocoles qui supportent le password change flow sans fournir l'ancien mot de passe, car par définition il est invalide ou inconnu.

La mécanique repose sur le protocole SAMR (Security Account Manager Remote Protocol). La séquence est :

  • Le client ouvre une session SAMR avec un null bind (session anonyme)
  • Il appelle SamrChangePasswordUser ou l'équivalent DCE/RPC en fournissant uniquement le nouveau mot de passe
  • Le DC accepte parce que le compte est déjà marqué expiré -aucune vérification de l'ancien mot de passe n'est effectuée

smbpasswd échoue ici car il tente un bind authentifié (il demande l'ancien mot de passe). changepasswd.py d'Impacket gère exactement ce cas : il détecte STATUS_PASSWORD_MUST_CHANGE, passe en null session SAMR, et pousse le nouveau mot de passe sans fournir l'ancien.

smbpasswd (échec) bind SAMR authentifié → demande old_password → NT_STATUS_LOGON_FAILURE changepasswd.py (succès) [!] Password is expired, trying to bind with a null session. null bind SAMR → SamrChangePasswordUser(new_password) → OK

Condition nécessaire : le DC doit autoriser les null sessions SAMR. C'est le comportement par défaut sur les DC Windows Server 2008+. Le flag RestrictAnonymousSAM = 0 (par défaut) ouvre ce chemin. 

changepasswd.py sendai.vl/thomas.powell:@dc.sendai.vl -newpass 'P@ssw0rd123!'

[!] Password is expired or must be changed, trying to bind with a null session.
[*] Connecting to DCE/RPC as null session
[*] Password was changed successfully.
nxc smb dc.sendai.vl -u 'thomas.powell' -p 'P@ssw0rd123!'

[+] sendai.vl\thomas.powell:P@ssw0rd123!

06 - gMSA Dump

gMSA Dump -mgtsvc$

Group Managed Service Accounts (gMSA) - comment fonctionne le dump

Un gMSA (msDS-GroupManagedServiceAccount) est un compte de service dont le mot de passe est généré automatiquement par le DC (256 octets aléatoires), renouvelé périodiquement, et stocké dans l'attribut LDAP msDS-ManagedPassword.

L'attribut msDS-GroupMSAMembership définit quels principals (utilisateurs ou groupes) sont autorisés à lire msDS-ManagedPassword. Si notre compte appartient à l'un de ces principals, on peut lire la structure MSDS-MANAGEDPASSWORD_BLOB directement via LDAP -pas d'exploit, c'est une lecture LDAP normale avec les bons droits.

La structure retournée contient :

  • Le mot de passe courant (NT hash calculable directement)
  • Le mot de passe précédent (pour la rotation)
  • Des clés Kerberos AES-128 et AES-256

gMSADumper.py effectue la lecture LDAP de msDS-ManagedPassword et calcule le NT hash à partir du blob. bloodyAD fait la même chose et affiche directement msDS-ManagedPassword.NT.

thomas.powell └─▶ membre de admsvc │ admsvc ∈ msDS-GroupMSAMembership de mgtsvc$ ▼ LDAP read: mgtsvc$.msDS-ManagedPassword │ décode MSDS-MANAGEDPASSWORD_BLOB ▼ NT hash de mgtsvc$ (sans changer aucun mot de passe)

Le NT hash d'un gMSA est utilisable directement pour Pass-the-Hash -les gMSA ne supportent pas l'authentification interactive mais fonctionnent parfaitement pour WinRM, SMB, et Kerberos. 

gMSADumper.py -u 'thomas.powell' -p 'P@ssw0rd123!' -d 'sendai.vl'

08 - Post-User Recon

Credentials in Service Registry

What pointed us here From the WinRM shell, service enumeration via the registry revealed a custom service with credentials hardcoded in its ImagePath value. A direct read of HKLM\SYSTEM\CurrentControlSet\services filtered on non-svchost executables surfaced it immediately.
Credentials dans les ImagePath de services - pourquoi ça arrive

L'attribut ImagePath d'un service Windows est la ligne de commande exécutée au démarrage du service. Certains services maison ou mal configurés passent des credentials directement sur la ligne de commande -typiquement avec des flags -u / -p -pour que le service puisse s'authentifier contre une ressource réseau au démarrage.

Ces valeurs sont stockées dans le registre et lisibles par n'importe quel utilisateur authentifié (les ACL par défaut sur HKLM\SYSTEM\CurrentControlSet\services donnent KEY_READ à BUILTIN\Users). Elles apparaissent en clair dans la valeur de type REG_EXPAND_SZ ou REG_SZ.

La commande PowerShell utilisée ici :

  • Lit toutes les sous-clés de services
  • Filtre celles contenant .exe mais pas svchost.exe (les services custom)
  • Cherche des patterns -u / -p dans les ImagePath
dir -Path HKLM:\SYSTEM\CurrentControlSet\services |
  Get-ItemProperty | Select-Object ImagePath |
  select-string -NotMatch "svchost.exe" | select-string "exe"

# Extrait utile parmi la sortie complète :
@{ImagePath=C:\WINDOWS\helpdesk.exe -u clifford.davey -p [redacted] -k netsvcs}

Un service custom nommé helpdesk passe les credentials de clifford.davey directement sur sa ligne de commande. Ces credentials sont valides contre SMB :

nxc smb dc.sendai.vl -u 'clifford.davey' -p '[redacted]'

[+] sendai.vl\clifford.davey:[redacted]
09 - Privilege Escalation

AD CS ESC4 to Administrator

What pointed us here clifford.davey est membre du groupe ca-operators. Ce groupe a des droits Full Control sur le template SendaiComputer. Certipy identifie cela comme ESC4 : des permissions dangereuses autorisent la réécriture du template, puis la demande d'un certificat en tant qu'Administrator.
AD CS ESC4 - réécriture de template

Dans AD CS, les templates de certificats sont des objets Active Directory stockés dans CN=Certificate Templates,CN=Public Key Services,CN=Services,CN=Configuration. Leurs attributs contrôlent qui peut enrôler, si le demandeur peut spécifier un SAN arbitraire (CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT), quels EKU sont inclus, etc.

ESC4 : un principal non-admin a des droits d'écriture sur un template (via WriteProperty, WriteDacl, WriteOwner ou GenericWrite/GenericAll). Ces droits donnent la main sur les attributs du template pour :

  • Activer CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT → le demandeur peut mettre n'importe quel UPN dans le SAN
  • Ajouter Client Authentication dans les EKU → le certificat servira à l'authentification Kerberos
  • Élargir les droits d'enrôlement → on peut s'auto-autoriser à enrôler

certipy template -write-default-configuration applique une configuration "ESC1-ready" : active le flag SAN, passe l'EKU en Client Auth, réduit la taille de clé minimale. Après ça, le template se comporte comme ESC1.

Avant (SendaiComputer original) msPKI-Certificate-Name-Flag : SubjectAltRequireDns ← SAN imposé par le DC msPKI-Enrollment-Flag : AutoEnrollment Enrollment Rights : Domain Computers, Domain Admins ↓ certipy template -write-default-configuration ↓ Après (SendaiComputer modifié) msPKI-Certificate-Name-Flag : 1 (ENROLLEE_SUPPLIES_SUBJECT) ← SAN libre msPKI-Enrollment-Flag : 0 pKIExtendedKeyUsage : 1.3.6.1.5.5.7.3.2 (Client Auth) → clifford.davey peut enrôler et fournir -upn administrator@sendai.vl

Pourquoi le premier certificat a échoué (SID mismatch)

Le premier certipy req sans -sid a produit un certificat sans extension SID. À l'authentification, le KDC a comparé l'UPN du certificat avec l'objet AD correspondant et a détecté que le SID de clifford.davey (qui a fait la demande) ne correspondait pas au SID d'Administrator → rejet avec Object SID mismatch.

La correction : passer -sid S-1-5-21-...-500 explicitement. Certipy inclut alors le SID Administrator dans une extension SAN URL et dans la security extension. Le KDC valide le SID embarqué, le mappe sur Administrator, et émet le TGT.

Sans -sid Certificat : SAN UPN = administrator@sendai.vl SID extension = (absente) KDC : UPN résout Administrator mais SID ≠ SID(clifford.davey) → REJECT Avec -sid S-1-5-21-3085872742-570972823-736764132-500 Certificat : SAN UPN = administrator@sendai.vl SAN URL SID = S-1-5-21-...-500 Security ext SID = S-1-5-21-...-500 KDC : SID correspond à Administrator → TGT émis pour Administrator

12 - Lessons

Key Takeaways

Concept Lesson
Null SMB partiel Même sans accès aux shares administratifs, une share accessible en lecture anonyme peut livrer une liste d'utilisateurs via les noms de répertoires.
STATUS_PASSWORD_MUST_CHANGE Ne pas confondre avec STATUS_LOGON_FAILURE. Un compte expiré peut être pris en main sans connaître l'ancien mot de passe via SAMR null bind -changepasswd.py gère ce cas là où smbpasswd échoue.
gMSA Un gMSA dont la msDS-GroupMSAMembership inclut un groupe accessible est une cible directe : lecture LDAP → NT hash → Pass-the-Hash. Pas d'exploit, juste des ACL mal positionnées.
Service ImagePath Les services custom passent parfois des credentials sur leur ligne de commande. Une lecture du registre HKLM\SYSTEM\CurrentControlSet\services filtré sur les exécutables non-svchost doit toujours faire partie de l'énumération post-shell.
ESC4 Des droits d'écriture sur un template AD CS suffisent à le transformer en ESC1. certipy template -write-default-configuration applique la configuration la plus permissive en une commande, puis le template devient exploitable comme ESC1.
SID mismatch (ESC4/ESC1) Sur un DC avec security extension activée, demander un certificat avec un UPN arbitraire sans fournir le SID correspondant produit Object SID mismatch. Le flag -sid de Certipy embarque explicitement le SID cible dans le certificat et contourne la vérification.
